## 0. 이 문서의 목적

메모를 쓰고 잊는 구조에서, **메모가 쌓일수록 AI가 똑똑해지는 구조**로 전환한다.

| 기존 방식 | 이 문서 이후 |
|-----------|-------------|
| 메모 → 폴더에 묻힘 → 잊음 | 메모 → 연결 → AI가 패턴 발견 → 새 아이디어 |
| 레퍼런스 수집 → 흩어짐 | 레퍼런스 → 태그 → 유사 작업 자동 추천 |
| 경험이 머릿속에만 존재 | 경험이 검색 가능한 DB로 축적 |

## LLM 위키란 — Karpathy의 개념

### 기존 방식(RAG) vs LLM 위키

```
기존 방식 (RAG — Retrieval-Augmented Generation):
  질문할 때마다 → 원본에서 조각 찾기 → 답변
  = 매번 도서관에서 책 찾아서 읽기
  
  문제:
  ├── 원본이 지저분하면 → 답변도 지저분
  ├── 연결 관계를 모름 (A와 B가 관련 있는지 모름)
  └── 지식이 축적되지 않음 (매번 처음부터)

LLM 위키:
  AI가 직접 위키를 쓰고, 관리하고, 업데이트
  = 사서가 도서관을 정리하고, 질문에 답하고, 새 책도 분류
  
  장점:
  ├── 원본이 지저분해도 → AI가 정리한 위키는 깨끗
  ├── 연결 관계를 AI가 직접 만듦
  └── 시간이 쌓일수록 위키가 풍부해짐
```

### 3-폴더 구조: /raw, /wiki, /schema

| 폴더 | 역할 | 규칙 | 비유 |
|------|------|------|------|
| `/raw` | 원본 자료 저장 | **절대 수정 금지** — 원본 그대로 보관 | 도서관 원본 서고 |
| `/wiki` | AI가 정리한 지식 | AI가 읽고 쓰고 업데이트 | 사서가 만든 요약 카드 |
| `/schema` | 정리 규칙 정의 | CLAUDE.md 또는 규칙 파일 | 도서관 분류 체계표 |

```
작동 흐름:

① 내가 /raw에 원본 저장 (회의록, 촬영 세팅, 레퍼런스, 메모)
              ↓
② AI가 /schema의 규칙에 따라 읽고 분석
              ↓
③ AI가 /wiki에 정리된 지식 문서를 생성/업데이트
              ↓
④ 내가 /wiki에 질문 → AI가 축적된 지식에서 답변
              ↓
⑤ 새 자료 추가 → /wiki 자동 업데이트 → 지식 누적
```

### 구체적 예시: /schema 규칙 파일

```markdown
# schema/촬영세팅_규칙.md

## 분류 기준
- /raw/촬영세팅/ 에 새 파일이 추가되면:
  1. 날씨 조건 추출 (맑음/흐림/실내)
  2. 조명 타입 분류 (자연광/스트로보/LED/혼합)
  3. 렌즈/ISO/조리개 값 추출
  4. 결과 만족도 기록 (있으면)

## 위키 업데이트 규칙
- /wiki/조명세팅_데이터베이스.md 에 행 추가
- 유사 조건의 기존 세팅이 있으면 비교 메모 추가
- 10건 이상 축적 시 → /wiki/조명세팅_패턴분석.md 자동 생성
```

## 왜 아티스트에게 중요한가

아티스트의 핵심 자산은 **경험 데이터**다. 문제는 이 데이터가 머릿속에만 있다는 것이다.

### 사진작가 — 촬영 세팅 패턴 발견

```
5년간 촬영 기록 1,200건을 /raw에 축적
→ AI가 /wiki에 패턴 분석 문서 생성

발견된 패턴:
┌─────────────────────────────────────────────────────────────┐
│ "흐린 날 야외 인물 촬영에서                                      │
│  ISO 400–800, f/2.8, 1/250을 가장 많이 사용했습니다.             │
│  이 조건에서 반사판 45도 각도일 때 고객 만족도가 가장 높았습니다."   │
│                                                               │
│ "자연광 인물 촬영의 고객 재의뢰율이 스튜디오 대비 2.3배 높습니다."   │
└─────────────────────────────────────────────────────────────┘

활용:
  다음 촬영 조건 입력 → AI가 최적 세팅 추천 (근거: 내 과거 데이터)
```

### 영상제작자 — 시놉시스 구조 분석

```
공모전 제출 시놉시스 20건을 /raw에 축적
→ AI가 당선작 vs 탈락작 비교 분석

발견된 패턴:
┌─────────────────────────────────────────────────────────────┐
│ "당선된 시놉시스의 공통 구조:                                    │
│  1) 로그라인이 반드시 1문장                                     │
│  2) 기획의도에 '개인 경험'이 포함                                │
│  3) 에피소드 구성이 3막이 아닌 4단계 (발견→몰입→전환→여운)"       │
│                                                               │
│ "탈락한 시놉시스의 공통 문제:                                    │
│  1) 로그라인이 3문장 이상                                       │
│  2) 예산 계획의 구체성 부족"                                     │
└─────────────────────────────────────────────────────────────┘

활용:
  새 공모전 공고 입력 → AI가 당선 패턴에 맞춘 시놉시스 구조 제안
```

### 공간기획자 — 공간 조건별 세팅 DB

```
전시 프로젝트 35건의 공간 조건 + 세팅 데이터 축적
→ AI가 조건별 최적 세팅 매트릭스 생성

"천장 5m + 면적 200평 + 프로젝터 3대"
→ 과거 유사 조건 3건에서 공통으로 사용한 세팅:
  프로젝터 배치(삼각), 조도 150lux, 벽면 반사율 고려 사항...
```

### 댄서 — 영감 패턴 추적

```
6개월간 인스타 영감노트 180건 축적
→ AI가 키워드 빈도 분석

"최근 3개월 영감노트에서 '바닥 동작'이 14회,
 '느린 전환'이 11회, '단색 조명'이 9회 등장.
 → 다음 작품 키워드 후보: 바닥 동작 × 느린 전환"
```

> **핵심: 시간이 쌓일수록 AI가 나를 더 잘 안다.** 1개월차와 12개월차의 AI는 완전히 다른 수준으로 답한다.

## 3-1. 왜 옵시디언인가

### 다른 도구와의 비교

| | 옵시디언 | 노션 | 에버노트 | 애플 메모 |
|---|---------|------|---------|----------|
| **파일 저장** | 내 컴퓨터 (마크다운) | 클라우드 (독자 형식) | 클라우드 | 클라우드 |
| **데이터 소유권** | ✅ 100% 내 파일 | ❌ 노션 서버 | ❌ 에버노트 서버 | ❌ iCloud |
| **AI 연동** | ✅ 플러그인으로 직접 연결 | ⚠️ 제한적 | ❌ | ❌ |
| **메모 간 연결** | ✅ `[[링크]]` 양방향 | ⚠️ 관계형 DB | ❌ | ❌ |
| **오프라인 사용** | ✅ 완전 | ❌ | ⚠️ | ✅ |
| **LLM 위키 구축** | ✅ 최적 | ❌ 파일 접근 불가 | ❌ | ❌ |
| **가격** | 무료 | 무료~유료 | 유료 | 무료 |

**옵시디언을 선택하는 이유 3가지:**

```
1. 파일이 내 컴퓨터에 있다
   → AI(Claude Code, Cursor)가 직접 파일을 읽고 쓸 수 있다
   → 노션은 불가능 (API 제한)

2. 마크다운 = 범용 포맷
   → 어떤 AI든 읽을 수 있다
   → 10년 후에도 열린다 (특정 앱에 종속 X)

3. [[링크]]로 메모가 연결된다
   → 시간이 지날수록 "나만의 위키"가 됨
   → AI에게 이 위키를 통째로 줄 수 있음
```

### 핵심 플러그인 4종

| 플러그인 | 기능 | 왜 필요한가 | 설치 난이도 |
|---------|------|-----------|-----------|
| **Smart Connections** | 노트 간 의미 유사도 분석 + AI 대화 | 내 메모 전체를 AI가 읽고, 관련 메모를 자동 추천 | ★★☆ |
| **Copilot** | 옵시디언 내 AI 어시스턴트 | 노트를 보면서 AI에게 바로 질문 가능 | ★★☆ |
| **Khoj** | 프라이버시 중시, 로컬 LLM 지원 | 데이터를 외부에 보내지 않고 AI 사용 가능 | ★★★ |
| **Web Clipper** | 웹 자료 원클릭 저장 | 레퍼런스를 옵시디언에 바로 저장 | ★☆☆ |

#### Smart Connections — 가장 중요한 플러그인

```
기능:
├── 현재 보고 있는 노트와 "의미적으로 유사한" 노트를 자동 추천
├── 옵시디언 내에서 AI와 대화 (내 노트 기반)
└── 노트 임베딩 → 벡터 검색

실전 사용:
  [[흐린날_야외촬영_0623]] 노트를 열면
  → 사이드바에 유사 노트 표시:
    - [[야외촬영_세팅_0510]] (유사도 92%)
    - [[비오는날_인물_0415]] (유사도 87%)
    - [[자연광_팁_정리]] (유사도 84%)

  AI 대화:
  나: "이 세팅과 비슷한 과거 촬영에서 가장 잘 나온 조건은?"
  AI: (내 노트 전체를 검색하여 답변)
```

#### Copilot — 노트 안에서 바로 질문

```
기능:
├── 옵시디언 사이드바에서 AI 채팅
├── 현재 노트 / 선택한 텍스트에 대해 질문
└── ChatGPT, Claude 등 다양한 LLM 연결 가능

실전 사용:
  회의록 노트를 열고 → Copilot에게:
  "이 회의록에서 액션 아이템만 뽑아줘"
  "이 내용을 클라이언트에게 보낼 요약으로 만들어줘"
```

#### Khoj — 프라이버시가 중요할 때

```
기능:
├── 로컬 LLM(Ollama 등)과 연결 가능
├── 데이터가 외부 서버로 나가지 않음
└── 셀프 호스팅 가능

언제 쓰나:
  - 클라이언트 계약서, 견적서 등 민감한 문서를 다룰 때
  - 외부 유출이 절대 안 되는 기업 프로젝트
```

#### Web Clipper — 레퍼런스 수집 자동화

```
기능:
├── 브라우저 확장 프로그램
├── 웹 페이지를 마크다운으로 변환 → 옵시디언에 저장
└── 템플릿 지정 가능 (자동 태그, 폴더 지정)

실전 사용:
  ① 레퍼런스 웹 페이지 발견
  ② Web Clipper 버튼 클릭
  ③ 옵시디언 /00_inbox/ 에 자동 저장
  ④ 나중에 태그 + [[링크]] 정리
```

### 학습 순서 — 이 순서를 지키지 않으면 포기한다

> 대부분의 포기 원인: 처음부터 플러그인을 10개 설치하고 → 복잡해지고 → 안 씀

| 주차 | 할 일 | 하지 말 것 |
|------|------|-----------|
| **1주차** | 옵시디언 설치 + 기본 사용 (메모, `[[링크]]`, 태그) | ❌ AI 플러그인 설치하지 마 |
| **2주차** | 매일 메모 1개 쓰기 습관화 → `00_inbox`에 넣기 | ❌ 폴더 구조 고민하지 마 (inbox만 쓰기) |
| **3주차** | Smart Connections 설치 → 노트 검색 + AI 대화 시작 | ❌ 다른 플러그인은 아직 |
| **4주차** | LLM 위키 구조 적용 (`/raw`, `/wiki`, `/schema`) | ❌ 완벽한 구조를 만들려 하지 마 |

#### 1주차 상세: 기본만

```
설치:
  obsidian.md → 다운로드 → 설치 → Vault(금고) 생성

이것만 익히기:
  1. 새 노트 만들기 (Ctrl/Cmd + N)
  2. [[링크]] 만들기 (대괄호 2개)
  3. #태그 달기
  4. 일일 노트 (Daily Note) 기능 켜기

매일 연습:
  아무 메모나 1개 쓰기
  → "오늘 촬영에서 배운 것"
  → "이 레퍼런스가 좋았던 이유"
  → "클라이언트 피드백 정리"
```

#### 2주차 상세: 습관 만들기

```
규칙 1개만:
  "메모는 00_inbox 폴더에 넣는다. 끝."

하루 5분:
  ① 하루 중 하나를 메모 (촬영 세팅, 아이디어, 회의 내용)
  ② [[링크]]로 관련 메모 연결 (있으면)
  ③ #태그 1~3개

7일 후 확인:
  inbox에 메모 7개 → "이것만으로 충분하다"
```

#### 3주차 상세: AI 연결

```
Smart Connections 설치:
  옵시디언 → 설정 → 커뮤니티 플러그인 → Smart Connections 검색 → 설치

사용법:
  ① 아무 노트 열기
  ② 오른쪽 사이드바 → Smart Connections 패널
  ③ 유사한 노트가 자동으로 표시됨
  ④ 채팅 탭에서 내 노트 기반으로 AI에게 질문

테스트 질문:

import os
from PIL import Image
from PIL.ExifTags import TAGS

def rename_photos(folder, project_name):
    photos = sorted([f for f in os.listdir(folder) if f.lower().endswith('.jpg')])
    for i, photo in enumerate(photos, 1):
        img = Image.open(os.path.join(folder, photo))
        exif = img._getexif()
        date = exif.get(36867, 'unknown').replace(':', '-')[:10]
        new_name = f"{date}_{project_name}_{i:03d}.jpg"
        os.rename(
            os.path.join(folder, photo),
            os.path.join(folder, new_name)
        )
        print(f"{photo} → {new_name}")

### 바이브코딩 — "코딩"이 들어가는 것은 전부 가능하다

> 파일 이름 바꾸기, 웹사이트 만들기 정도가 아니다.
> **실제 작업 프로세스를 자동화하는 도구**를 만들 수 있다.

**바이브코딩이란**:
```
"분위기(vibe)로 코딩한다"
= 내가 원하는 걸 말로 설명하면 AI가 코드를 짜고 실행한다
= 코드를 읽을 필요 없다
= 작동하면 된다
```

### 실전 예시: 조명 큐시트 자동화

```
기존 방식:
  음악 듣기 → 구간 나누기 → 엑셀에 시간/조명값 수기 입력
  → 수정 → 또 수정 → 2시간

바이브코딩:
  "이 음악 파일 분석해서 조명 큐시트 만들어줘.
   - BPM 변화 구간 자동 감지
   - 각 구간에 조명 색상/밝기 매칭
   - DMX 채널값까지 출력
   - 엑셀 or CSV로 내보내기"

  → AI가 Python 스크립트 생성 → 실행
  → 음악 분석 → 큐시트 자동 생성 → 5분
```

**여기서 끝이 아니다**:
```
"이 큐시트를 MA2/GrandMA 형식으로 변환해줘"
→ 실제 조명 콘솔에 올릴 수 있는 파일 생성

"리허설 때 수정한 부분 반영해줘. 2분 30초~3분 구간 빨간색→파란색"
→ 큐시트 자동 업데이트

"다음 공연 음악 바꿨어. 새 음악으로 비슷한 패턴 적용해줘"
→ 이전 큐시트 패턴 학습 → 새 음악에 자동 적용
```

### 아티스트가 바이브코딩으로 만들 수 있는 것들

| 직군 | 만들 수 있는 도구 | 어떻게 시키나 |
|------|-----------------|-------------|
| **사진작가** | 촬영 조건 입력 → 세팅 추천기 | "ISO/조리개/셔터 조합을 조건별로 추천하는 웹앱 만들어줘" |
| **사진작가** | 사진 셀렉 보조 (흔들림/노출 자동 필터) | "이 폴더 사진에서 흔들린 것, 노출 오버인 것 자동 걸러줘" |
| **영상제작자** | 자막 타임코드 자동 생성기 | "이 영상 음성 분석해서 SRT 자막 파일 만들어줘" |
| **영상제작자** | 촬영 분량 계산기 | "시놉시스 기반으로 예상 촬영 일수/장비/예산 계산해줘" |
| **공간기획자** | 공간 배치 시뮬레이터 | "이 평면도에 작품 30점 배치 시뮬레이션 만들어줘" |
| **공간기획자** | 예산 자동 산출기 | "공간 면적/장비 수량 넣으면 예산 자동 계산되는 도구" |
| **댄서** | 음악 BPM/구간 분석기 | "이 음악의 BPM 변화 구간 시각화해줘" |
| **댄서** | 리허설 타이머 | "큐시트 기반으로 구간별 알림 주는 타이머 만들어줘" |

### 바이브코딩의 핵심: 반복되면 → 도구로 만들어라

```
1회성 작업 → 그냥 AI에게 시키면 됨
                (채팅에서 "해줘")

반복 작업   → 도구를 만들어서 매번 쓴다
                ("이런 도구 만들어줘" → 계속 재사용)
```

```
예시:
  견적서를 매번 새로 만든다 → 견적서 생성 도구를 한 번 만든다
  큐시트를 매번 엑셀로 짠다 → 큐시트 생성기를 한 번 만든다  
  포트폴리오를 매번 갱신한다 → 자동 갱신 시스템을 한 번 만든다
```

**필요한 것: Cursor 또는 Antigravity (설치형)**
**필요 없는 것: 코딩 지식**
**하는 법: 말로 설명하면 됨**

### 더 열어두기 — "이것도 된다"

> 위의 예시들은 시작일 뿐이다.
> 아래는 **실제로 가능한 것들**이다. 지금 당장 할 필요는 없다.
> 하지만 "이게 된다"는 걸 알아두면, 필요한 순간에 AI에게 시킬 수 있다.

#### 하드웨어 제어

```
"아두이노로 관객 동선 센서 달아서, 
 사람이 지나가면 조명이 반응하는 인터랙티브 설치 만들어줘"

"라즈베리파이에 카메라 달아서,
 전시장 관객 수 실시간 카운트하는 시스템 만들어줘"

→ 회로 설계 + 코드 + 배선도까지 AI가 생성
→ 부품 구매 목록 + 예상 비용까지 산출
```

#### 실시간 비주얼 / 공연 연동

```
"TouchDesigner 패치 만들어줘.
 - 음악 입력 받아서 실시간 비주얼 생성
 - BPM에 파티클 반응
 - MIDI 컨트롤러로 색상 전환"

"공연 중 댄서의 움직임을 카메라로 추적해서
 배경 영상이 실시간으로 반응하게 해줘"

→ MediaPipe 포즈 추정 + 프로젝션 매핑 연동
```

#### 나만의 앱 / 도구

```
"클라이언트가 촬영 날짜/유형/장소 선택하면
 자동으로 견적+일정+준비물 리스트 나오는 웹앱 만들어줘"

"내 포트폴리오를 보고 클라이언트가 직접 사진 골라서
 다운로드할 수 있는 갤러리 시스템 만들어줘"

"전시 관람객이 QR 찍으면 AR로 작품 설명 뜨게 해줘"
```

#### 영상/사운드 자동화

```
"이 영상 30개에서 하이라이트 구간 자동 추출해서
 쇼릴 편집해줘"

"이 인터뷰 영상에서 '음', '어' 같은 필러 자동 제거해줘"

"이 댄스 영상 10개 분석해서
 동작 전환 타이밍 패턴 시각화해줘"
```

#### 모션 캡처 / 포즈 분석

```
"이 연습 영상에서 내 포즈를 프레임별로 추출해줘"

"두 영상(레퍼런스 vs 내 연습) 비교해서
 자세 차이 나는 부분 표시해줘"

→ MediaPipe/OpenPose 기반, 카메라만 있으면 됨
→ 모캡 장비 수백만원 → 웹캠 하나로 대체
```

#### 커스텀 플러그인

```
"라이트룸에서 내가 자주 쓰는 보정 순서를
 원클릭 프리셋으로 만들어줘"

"프리미어 프로에서 자막 스타일 일괄 적용하는
 스크립트 만들어줘"

"블렌더에서 공간 배치를 CSV로 불러와서
 자동 배치하는 애드온 만들어줘"
```

#### 비즈니스 자동화

```
"인스타 DM으로 문의 오면
 자동 응답 + 견적 폼 링크 보내는 봇 만들어줘"

"촬영 끝나면 클라이언트에게
 갤러리 링크 + 피드백 폼 자동 발송되게 해줘"

"매월 매출/지출 자동 집계해서
 대시보드로 보여주는 시스템 만들어줘"
```

#### 제너레이티브 아트

```
"내 사진 스타일을 학습해서
 비슷한 느낌의 배경 패턴 무한 생성하는 코드 만들어줘"

"관객 트위터 실시간 감정 분석해서
 전시장 벽면 색상이 바뀌는 설치 만들어줘"
```

> **핵심: "이게 된다고?" → "된다."**
> 직접 코드를 짤 필요 없다. AI에게 말로 시키면 된다.
> 당장 전부 할 필요도 없다.
> **필요한 순간에 "이게 가능하다"는 걸 아는 것** 자체가 확장이다.

## 3-5. 실전 사례: 실제 동작하는 LLM 위키 시스템

> **이건 이론이 아니다.** 아래는 실제로 매일 사용되고 있는 LLM 위키 시스템의 구조다. 이런 것을 AI에게 "만들어줘"라고 하면 만들어준다.

### 실제 시스템의 3-레이어 구조

```
REAL_HAEPA/                          ← 옵시디언 Vault = 전체 지식 시스템
│
├── 01_Inbox/                        ← 레이어 1: 원본 (Read-Only)
│   └── (메모, 기사, 웹 클립 등)       AI가 절대 수정 안 함. 원본 그대로 보관.
│
├── concepts/                        ← 레이어 2-A: 허브 노드 (AI가 소유)
│   ├── 그럼에도_불구하고.md            개념/테마/프레임워크
│   ├── site_brief.md                 프로젝트 기획서
│   └── site_content_spec.md          페이지별 상세 명세
│
├── pairings/                        ← 레이어 2-B: 페어링 (AI가 소유)
│   ├── 2026-04-19_성난사람들.md        콘텐츠 × 이론 분석
│   ├── 2026-04-20_만약에우리.md        날짜_주제.md 형식
│   └── 2026-04-22_EEAAO_파우스트.md
│
├── 02_Projects/                     ← 레이어 3: 프로젝트 작업 공간
│   ├── ai-education/                 실제 프로젝트 폴더
│   ├── curation-engine/
│   └── detective-system-webnovel/
│
├── index.md                         ← 전체 위키 카탈로그
├── log.md                           ← 변경 이력 (append-only)
├── wiki.py                          ← CLI 도구
└── CLAUDE.md                        ← AI 행동 규칙 (스키마)
```

앞에서 설명한 `/raw`, `/wiki`, `/schema` 3-폴더 구조가 실제로는 이렇게 구현된다:

| 개념 | 실제 구현 | 역할 |
|------|----------|------|
| `/raw` | `01_Inbox/` | 원본 자료. AI는 읽기만 함 |
| `/wiki` | `concepts/` + `pairings/` | AI가 직접 쓰고 관리하는 지식 |
| `/schema` | `CLAUDE.md` | AI의 행동 규칙과 템플릿 정의 |

### 실제 워크플로우 3종: INGEST → QUERY → LINT

이 시스템에는 3개의 워크플로우가 정의되어 있다. CLAUDE.md에 규칙으로 적혀 있고, AI는 이 규칙을 따른다.

#### INGEST — 새로운 영감을 흡수할 때

```
사용자: "이 영화 봤어. 01_Inbox에 메모 넣었으니 처리해줘"

AI 동작 순서:
  ① 01_Inbox/ 의 원본 파일 읽기
  ② 사용자와 대화: "핵심이 뭔가요? 기존 개념과 연결할까요?"
  ③ pairings/ 에 새 파일 생성 (페어링 템플릿 적용)
  ④ 관련 concepts/ 허브 노드에 역참조 추가
  ⑤ index.md 에 새 항목 추가
  ⑥ log.md 에 기록 남기기

핵심: 사용자는 메모를 inbox에 던지기만 하면 된다.
      나머지는 AI가 규칙에 따라 자동 처리.
```

#### QUERY — 지식을 탐색할 때

```
사용자: "내가 지금까지 '허무주의 극복'에 대해 모은 자료 정리해줘"

AI 동작 순서:
  ① index.md 먼저 읽기 → 관련 페이지 식별
  ② concepts/ 와 pairings/ 의 해당 파일 읽기
  ③ 인용 기반으로 종합 답변 생성
  ④ 새로운 연결 발견 시 → "이걸 새 개념 페이지로 만들까요?"
  ⑤ 사용자 승인 → 새 페이지 생성 → index + log 업데이트

핵심: 쌓인 지식이 많을수록 답변 품질이 올라간다.
```

#### LINT — 위키 건강 검진

```
AI 자동 점검 항목:
  ① 깨진 링크: [[존재하지_않는_페이지]] 발견 → 보고
  ② 고아 페이지: 아무 데서도 연결되지 않은 파일 발견 → 보고
  ③ 의미 충돌: 같은 개념이 다른 페이지에서 다르게 정의 → 경고
  ④ 용어 불일치: 같은 것을 다른 이름으로 부르는 경우 → 통일 제안

결과:
  "깨진 링크 2건, 고아 페이지 1건 발견. 수정할까요?"
```

### wiki.py — AI에게 만들어달라고 한 CLI 도구

이 시스템에는 `wiki.py`라는 CLI 도구가 포함되어 있다. 이 코드를 직접 짠 것이 아니라, **AI에게 "위키 관리용 CLI 도구 만들어줘"라고 시켜서 만든 것**이다.

```bash
# 위키 파일 목록
python wiki.py list

# 텍스트 검색
python wiki.py search "조명"

# 건강 검진 (깨진 링크, 고아 페이지)
python wiki.py lint

# 통계
python wiki.py stats
```

실제 `wiki.py`의 주요 기능:

| 명령어 | 동작 | 실제 코드가 하는 일 |
|--------|------|-------------------|
| `list` | 전체 파일 목록 | `concepts/` + `pairings/` 폴더를 스캔해서 색상 구분 출력 |
| `search <query>` | 텍스트 검색 | 모든 .md 파일에서 쿼리를 검색, 라인 번호와 하이라이트 출력 |
| `lint` | 위키 무결성 검사 | `[[링크]]`를 정규식으로 파싱 → 실제 파일 존재 여부 확인 → 깨진 링크/고아 페이지 보고 |
| `stats` | 통계 | 총 페이지 수, 개념/페어링 수, 총 링크 수, 페이지당 평균 링크 수 |

```
출력 예시 (stats):

=== LLM-Wiki Statistics ===
  Total Pages:      6
    - Concepts:     3
    - Pairings:     3
  Total Links:      12
  Avg Links/Page:   2.00
```

**이 도구의 핵심**: 코드를 이해할 필요 없다. "위키 파일 목록 보여주는 CLI 도구 만들어줘. 검색이랑 링크 검사 기능도 넣어줘"라고 AI에게 시키면 이런 도구가 나온다.

### index.md — 위키 네비게이션

이 시스템은 `index.md`라는 파일로 전체 위키를 탐색한다. 모든 INGEST/QUERY 작업 후 자동 업데이트된다:

```markdown
# 🏛️ LLM-Wiki Index

## 🌌 1. 개념 및 허브 노드 (Concepts)
- [[그럼에도_불구하고]] — 허무·좌절을 확인한 뒤에도 긍정하는 태도
- [[site_brief]] — 포트폴리오 사이트 기획 브리프
- [[site_content_spec]] — 페이지별 상세 명세

## 📽️ 2. 콘텐츠 및 사유 페어링 (Pairings)
- [[2026-04-19_성난사람들]] — K-이미지 담론, 테크노 오리엔탈리즘
- [[2026-04-20_만약에우리]] — 관계 속 희생과 구원의 허구성
- [[2026-04-22_EEAAO_파우스트]] — 세속 시대의 실존적 용기
```

AI에게 질문하면, AI는 **index.md를 먼저 읽고** → 관련 파일을 찾아서 → 깊이 읽는다. 파일이 100개가 되어도 index가 목차 역할을 하기 때문에 검색이 빠르다.

### 아티스트 적용 — 이 구조를 촬영/전시/공연에 쓰면

이 위키 시스템은 영화/책 분석용이지만, 구조 자체는 어떤 분야에든 그대로 적용된다:

| 원래 시스템 | 사진작가 적용 | 공간기획자 적용 |
|------------|-------------|---------------|
| `01_Inbox/` (영화 메모) | `01_Inbox/` (촬영 후 메모, 클라이언트 브리프) | `01_Inbox/` (공간 답사 메모, RFP 원본) |
| `concepts/` (철학 개념) | `concepts/` (조명기법, 장비세팅, 보정스타일) | `concepts/` (공간유형, 프로젝터배치, 동선설계) |
| `pairings/` (영화 × 이론) | `pairings/` (촬영세팅 × 결과, 레퍼런스 × 분석) | `pairings/` (전시사례 × 공간조건, 작가 × 설치방식) |
| `CLAUDE.md` (분석 규칙) | `CLAUDE.md` (촬영 기록 규칙, 세팅 분류 기준) | `CLAUDE.md` (공간 기록 규칙, 장비 스펙 양식) |
| `index.md` (전체 목차) | `index.md` (전체 촬영/장비/기법 목차) | `index.md` (전시 프로젝트/공간/장비 목차) |
| `wiki.py` (검색/점검) | `wiki.py` (세팅 검색, 태그 점검) | `wiki.py` (공간 검색, 장비 목록 점검) |

#### 사진작가 페어링 예시

```markdown
# [26/06/23] 브랜드A 야외촬영, 흐린날 자연광, 인물 전신

## 촬영 후 메모 — 내 관찰
오후 3시, 구름 80%. 반사판 없이 촬영했는데 얼굴에 그림자가 안 생겨서 오히려 좋았다.
85mm f/1.4에서 배경 분리가 과했음. f/2.0이 더 나았을 듯.

---

## 세팅 분석 — 이 결과를 설명할 수 있는 기술 요소
### 1. 확산광 원리
**핵심**: 흐린 날 구름 = 거대한 소프트박스
**논리**: 광원이 클수록 그림자가 부드러움
**내 관찰과의 연결**: 반사판 없이도 그림자가 안 생긴 이유

---

## 다음에 시도할 것
- f/2.0 vs f/1.4 비교 촬영
- 흐린 날 + 반사판 유무 비교

---

## 연결
- 유사 촬영: [[2026-05-10_잡지B_야외]]
- 관련 개념: [[자연광_인물촬영]], [[조리개_배경분리]]
```

#### 사진작가용 허브 노드 예시

### AI에게 위키 시스템을 만들어달라고 하는 방법

```
실제로 하면 되는 것:

① 옵시디언 Vault를 만든다

② AI에게 이렇게 말한다:
   "이 Vault를 LLM 위키로 관리해줘.
    - 01_Inbox에 넣은 자료를 읽고 정리해서 wiki 폴더에 써줘
    - 이 템플릿을 따라줘: [위의 페어링 템플릿 붙여넣기]
    - index.md를 항상 업데이트해줘
    - CLAUDE.md에 이 규칙을 넣어줘"

③ AI에게 CLI 도구도 시킨다:
   "위키 파일 검색하고, 깨진 링크 점검하는 CLI 도구 만들어줘"

④ 2주 쓰면 AI가 내 패턴을 파악한다.
   4주 쓰면 "이 촬영은 저번 건과 비슷한데, 그때 세팅을 참고하세요"라고 말한다.
```

## 요약 — 이 문서의 핵심

```
메모 앱     →  옵시디언 (연결 가능한 메모)
파일 보관   →  LLM 위키 (AI가 관리하는 지식 시스템)
경험        →  검색 가능한 데이터 (시간이 쌓일수록 강해짐)
반복 작업   →  AI에게 코딩 시키기 (코딩을 배울 필요 없음)
```

| 단계 | 하는 일 | 소요 시간 |
|------|---------|----------|
| 1단계 | 옵시디언 설치 + 기본 사용 | 1주 |
| 2단계 | 매일 메모 습관 | 2주 |
| 3단계 | Smart Connections로 AI 연결 | 3주 |
| 4단계 | LLM 위키 구조 적용 | 4주 |
| 5단계 | 코딩 자동화 (필요할 때마다) | 지속 |

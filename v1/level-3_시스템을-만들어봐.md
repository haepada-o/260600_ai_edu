## 0. 이 문서는 누구를 위한 것인가

이 문서는 **AI를 "도구"가 아니라 "시스템"으로 만들고 싶은 사람**을 위한 것입니다.

| 조건 | 설명 |
|------|------|
| ✅ 레벨 2를 마친 사람 | 규칙 파일, 커스텀 명령어를 이미 사용 중 |
| ✅ AI를 일상적으로 업무에 쓰고 있는 사람 | 하루에 여러 번 AI와 대화한다 |
| ✅ 더 자동화하고 싶은 사람 | "이건 AI가 알아서 해줬으면" 싶은 일이 있다 |

> **"재미없는 일을 AI한테 최대한 매뉴얼화해서 시켜야 돼요."**

**이 문서를 마치면 할 수 있는 것:**
- 🤖 역할별 에이전트를 설계하고 배치할 수 있다
- 📋 반복 업무를 SOP(표준 절차)로 정의할 수 있다
- 🔍 AI가 스스로를 검증하는 루프를 만들 수 있다
- 📚 시간이 지날수록 더 똑똑해지는 지식 시스템을 구축할 수 있다
- 🔗 여러 에이전트를 연결한 파이프라인을 설계할 수 있다

> [!NOTE]
> 레벨 1에서는 **"어떻게 말하느냐"** (대화 기술)  
> 레벨 2에서는 **"어떻게 기억시키느냐"** (규칙과 도구)  
> 레벨 3에서는 **"어떻게 시스템을 만드느냐"** (설계와 자동화)

## 1. 에이전트: AI에게 역할을 준다

### 핵심 개념

> **하나의 AI에게 모든 걸 시키지 말고, 역할별로 나눠라.**

직장에서도 한 사람이 리서치, 기획, 작성, 검토, 발표를 다 하면 품질이 떨어집니다.  
AI도 마찬가지입니다.

| 방식 | 비유 | 결과 |
|------|------|------|
| ❌ 하나의 AI에게 전부 | 인턴 1명에게 모든 일 | 두루뭉술한 결과 |
| ✅ 역할별 에이전트 | 전문가 팀 구성 | 각 분야에서 높은 품질 |

---

### 실제 사례 — B-Square 10종 에이전트 시스템

한 자산운용사에서 실제로 운영 중인 에이전트 체계입니다.

#### 범용 에이전트 7종

어떤 업무에서든 공통으로 사용하는 에이전트:

| 에이전트 | 역할 | 언제 쓰나 | 예시 요청 |
|----------|------|----------|----------|
| **Researcher** | 심층 리서치 | 공고 분석, 정책 조사, 경쟁사 분석 | "이번 해양 펀드 공고의 핵심 요구사항 정리해줘" |
| **Analyst** | 데이터 분석 | 배점 시뮬레이션, 재무 분석 | "이 포트폴리오의 IRR 시나리오 3개 돌려줘" |
| **Writer** | 문서 작성 | 제안서, 보고서, 기획안 | "투자 심사 보고서 초안 작성해줘" |
| **Reviewer** | 검토/교정 | 정합성 확인, 교정 | "이 제안서에서 수치 불일치 찾아줘" |
| **PR Agent** | 홍보/커뮤니케이션 | 보도자료, SNS | "이 투자 건 보도자료 작성해줘" |
| **Monitor** | 자동 모니터링 | 뉴스, 공고, 트렌드 수집 | "이번 주 부동산 펀드 공고 모아줘" |
| **Report Agent** | 리포트 생성 | LP 보고서, 맞춤 리포트 | "3분기 LP 보고서 만들어줘" |

#### 도메인 에이전트 3종

특정 업무 영역에 특화된 에이전트:

| 에이전트 | 역할 | 범위 |
|----------|------|------|
| **Fund Agent** | 펀드 출자 전 과정 | 공고분석 → 기획안 → 제안서 → 구술심사 준비 |
| **Ops Agent** | 수주 후 사업관리 | 프로젝트 관리, 일정, 산출물 관리 |
| **Valuup Agent** | 밸류업파트너 전용 | 234개 피투자기업 성장 지원 관리 |

> [!IMPORTANT]
> **핵심 포인트**: 범용 에이전트는 **어떤 일을 하느냐**로 나누고,  
> 도메인 에이전트는 **어떤 분야에서 하느냐**로 나눕니다.  
> 도메인 에이전트는 필요할 때 범용 에이전트를 호출합니다.

```
예시 흐름:

Fund Agent가 새 펀드 공고를 발견
  → Researcher에게 "공고 분석해줘"
  → Analyst에게 "배점 시뮬레이션 돌려줘"
  → Writer에게 "제안서 초안 써줘"
  → Reviewer에게 "제안서 검토해줘"
```

### 에이전트 설계 템플릿

자신만의 에이전트를 만들 때 이 템플릿을 사용하세요:

```markdown
# [에이전트 이름]

## 역할
[한 문장으로 정의]

## 언제 호출하나
- [상황 1]
- [상황 2]
- [상황 3]

## 핵심 원칙
1. [원칙 1]
2. [원칙 2]
3. [원칙 3]

## 참조 파일
- [파일 경로 1: 무엇인지]
- [파일 경로 2: 무엇인지]

## 출력 양식
[이 에이전트가 만드는 결과물의 형식]

## 금지 사항
- [하지 말 것 1]
- [하지 말 것 2]
```

#### 실전 예시: Researcher 에이전트

```markdown
# Researcher

## 역할
주어진 주제에 대해 심층 리서치를 수행하고, 구조화된 분석 결과를 제공한다.

## 언제 호출하나
- 새로운 펀드 공고가 나왔을 때
- 정책/규제 변화를 파악해야 할 때
- 경쟁사 동향을 분석해야 할 때
- 투자 대상 시장을 조사해야 할 때

## 핵심 원칙
1. 사실과 의견을 명확히 구분한다
2. 모든 수치에 출처를 밝힌다 (기관명 + 발표 시점)
3. "~로 보입니다" 같은 애매한 표현 대신 근거 기반 판단을 제시한다
4. 리서치 범위와 한계를 명시한다

## 참조 파일
- /data/industry-reports/: 업계 보고서 아카이브
- /data/regulations/: 관련 법규 및 정책 문서

## 출력 양식
1. 핵심 요약 (3줄 이내)
2. 상세 분석 (구조화된 섹션)
3. 데이터 표 (있을 경우)
4. 시사점 및 제안
5. 출처 목록

## 금지 사항
- 출처 없는 수치 제시 금지
- "일반적으로", "보통" 같은 근거 없는 일반화 금지
- 분석 없이 자료 나열만 하는 것 금지
```

---

### 에이전트를 만드는 실전 팁

> [!TIP]
> **에이전트 설계 3원칙:**
> 1. **역할은 한 문장**: "이 에이전트는 ___를 한다"로 설명 안 되면 역할이 너무 넓다
> 2. **금지 사항이 핵심**: AI에게 "하지 마라"를 알려주는 것이 "해라"보다 중요하다
> 3. **출력 양식 고정**: 결과물의 형태를 정해놔야 다음 에이전트가 받아서 쓸 수 있다

**처음 만들 에이전트 추천:**

| 직군 | 첫 번째 에이전트 | 이유 |
|------|----------------|------|
| 사무직 | **이메일 작성기** | 가장 자주, 가장 반복적 |
| 기획자 | **리서치 정리기** | 자료 정리가 가장 시간 소모 |
| 영업 | **제안서 검토기** | 실수 방지 효과가 가장 큼 |
| 마케터 | **콘텐츠 톤 검수기** | 브랜드 일관성 유지 |
| 관리자 | **보고서 초안기** | 정기 보고서 작성 시간 절약 |

## 2. 스킬(SOP): 반복 과업을 절차로 정의한다

### 핵심 개념

> 에이전트가 **"누구"**라면, 스킬은 **"무엇을 어떻게"**이다.

| 개념 | 비유 | 예시 |
|------|------|------|
| 에이전트 | 직책 | "당신은 투자 분석가입니다" |
| 스킬(SOP) | 업무 매뉴얼 | "투자 심사 보고서는 이 순서로 씁니다" |

에이전트만 있으면: "분석해줘" → (뭘 어떻게?) → 매번 다른 결과  
에이전트 + 스킬이 있으면: "분석해줘" → (SOP대로) → 일관된 결과

---

### 실제 사례 — 15종 스킬 시스템

실제 운영 중인 스킬 체계입니다. **스킬이 체인처럼 연결**되는 것에 주목하세요:

```
📋 공고분석 → 📝 기획안 → 📄 펀드출자제안서 → 🎤 펀드구술심사 → 📊 발표자료자동화
```

#### 전체 스킬 목록

| # | 스킬 이름 | 입력 | 산출물 | 다음 스킬 |
|---|----------|------|--------|----------|
| 1 | **공고분석** | 펀드 공고문 | 핵심 요구사항 + 배점 분석표 | → 기획안 |
| 2 | **기획안** | 공고분석 결과 | 대응 전략 + 구조 기획안 | → 펀드출자제안서 |
| 3 | **펀드출자제안서** | 기획안 + 사내 데이터 | 완성된 제안서 초안 | → QA체크리스트 |
| 4 | **QA체크리스트** | 제안서 초안 | 체크 결과 + 수정 사항 | → 펀드구술심사 |
| 5 | **펀드구술심사** | 제안서 최종본 | 예상 질문 + 답변 스크립트 | → 모의심사 |
| 6 | **모의심사** | 심사 스크립트 | 피드백 + 취약점 분석 | → 발표자료자동화 |
| 7 | **발표자료자동화** | 제안서 + 심사 준비 | 발표 슬라이드 구성안 | - |
| 8 | **LP보고서** | 펀드 운용 데이터 | 분기 LP 보고서 | - |
| 9 | **투자심사보고서** | 투자 대상 정보 | 투자 심사 보고서 | - |
| 10 | **밸류업리포트** | 기업 현황 데이터 | 밸류업 성과 보고서 | - |
| 11 | **보도자료** | 뉴스거리 + 팩트 | 보도자료 초안 | - |
| 12 | **경쟁사분석** | 경쟁사 목록 | 비교 분석표 + SWOT | - |
| 13 | **정책분석** | 정책/규제 문서 | 영향 분석 + 대응 방안 | - |
| 14 | **뉴스모니터링** | 키워드 목록 | 일간/주간 뉴스 브리핑 | - |
| 15 | **회의록정리** | 회의 녹음/메모 | 구조화된 회의록 + 액션아이템 | - |

> [!TIP]
> **스킬 체인의 위력**: 스킬 1~7이 연결되면, **공고 하나가 나왔을 때 발표 준비까지  
> 하나의 흐름으로 처리**할 수 있습니다. 이것이 바로 "파이프라인"입니다 (5장에서 자세히).

### 스킬 SOP 템플릿

```markdown
# [스킬 이름] SOP

## 목적
[이 스킬이 만드는 산출물은 무엇인가]

## 선행 조건
- [필요한 입력 데이터]
- [참조할 규칙 파일]
- [이전 스킬의 산출물 (있다면)]

## 단계
1. [Step 1: 무엇을, 어떻게]
   - 세부 지침
   - 주의 사항
2. [Step 2: 무엇을, 어떻게]
   - 세부 지침
   - 주의 사항
3. [Step 3: ...]
4. ...

## 품질 기준
- [체크 항목 1: 예) 모든 수치에 출처가 있는가]
- [체크 항목 2: 예) 요구사항 누락 없는가]
- [체크 항목 3: 예) 양식에 맞는가]

## 다음 스킬
- [이 스킬 다음에 실행할 스킬]
- [필요한 경우 분기 조건]
```

#### 실전 예시: 공고분석 SOP

```markdown
# 공고분석 SOP

## 목적
펀드 출자 공고의 핵심 요구사항을 추출하고, 배점 기반 전략을 수립한다.

## 선행 조건
- 공고문 원본 (PDF 또는 텍스트)
- 과거 유사 공고 대응 기록 (있다면)

## 단계
1. 공고 기본 정보 추출
   - 공고기관, 펀드 규모, 출자 규모, 마감일
   - GP 자격 요건 (인가, 트랙레코드 등)
   
2. 배점표 분석
   - 각 평가 항목별 배점 추출
   - 배점 비중이 높은 항목 하이라이트 (상위 3개)
   - 정량 vs 정성 평가 비중 파악

3. 핵심 요구사항 정리
   - 필수 요건 (미충족 시 탈락)
   - 가점 요건 (차별화 포인트)
   - 감점 요건 (주의 사항)

4. 자사 적합도 평가
   - 각 요구사항 대비 자사 현황 매칭
   - 강점/약점 도출
   - 준비 필요 사항 리스트

5. 전략 제안
   - 강점 부각 포인트
   - 약점 보완 방안
   - 차별화 전략

## 품질 기준
- □ 배점표의 모든 항목이 빠짐없이 추출되었는가
- □ 필수 요건이 명확히 구분되었는가
- □ 자사 적합도 판단에 근거가 있는가
- □ 전략 제안이 구체적인가 (vs "잘 준비하자" 수준)

## 다음 스킬
- → 기획안 SOP (적합도 판단이 "참여"인 경우)
- → 종료 (적합도 판단이 "불참"인 경우, 사유 기록)
```

## 3. 검증 루프: 시스템이 스스로를 검증한다

### 문제: AI는 "좋습니다!"라고만 한다

AI에게 글을 보여주면 대부분 이렇게 답합니다:

```
❌ 전형적인 AI 반응:
"좋은 글이네요! 몇 가지 작은 수정만 하면 완벽할 것 같습니다."
"전체적으로 잘 쓰셨습니다. 일부 표현을 다듬으면 더 좋겠네요."
```

이건 **검증이 아닙니다.** 칭찬이죠.  
진짜 검증은 **문제를 찾아내는 것**입니다.

---

### 실제 사례 — 웹소설 프로젝트의 "악플러" 에이전트

한 웹소설 프로젝트에서는 이 문제를 **역할 분리**로 해결했습니다.

> **"너는 악플러다. 칭찬은 한 줄도 쓰지 마라. 까는 게 일이다."**

이 에이전트의 규칙:

```markdown
# 악플러 에이전트

## 역할
작품의 약점을 찾아내고 파괴적으로 비평한다.

## 핵심 규칙
- 칭찬 금지. 단 한 줄도.
- "좋다", "잘했다", "흥미롭다" 등 긍정 표현 사용 불가
- 발견한 문제만 보고한다
- 각 문제에 구체적 근거를 제시한다
```

#### 8가지 사냥 목록

이 악플러 에이전트가 찾아내는 8가지 문제 유형:

| # | 사냥 대상 | 설명 | 예시 |
|---|----------|------|------|
| 1 | **클리셰 사냥** | 이미 어디서 본 전개 | "주인공이 위기에서 숨겨진 힘 각성 → 수백 번 본 전개" |
| 2 | **동기 부재** | 캐릭터가 왜 이걸 하는지 설명 없음 | "갑자기 목숨을 건다고? 이유가 없는데?" |
| 3 | **편의적 전개** | 작가 편한 대로 흘러감 | "딱 필요한 순간에 딱 필요한 정보가 등장" |
| 4 | **감정 인플레** | 매번 울고 감동하면 가치 하락 | "5화 연속 눈물 장면 → 독자는 이미 무감각" |
| 5 | **설명충** | 보여주지 않고 설명만 함 | "'그는 슬펐다' 대신 행동으로 보여줘" |
| 6 | **동어반복** | 같은 말을 다르게 반복 | "결국 3문단이 같은 말을 하고 있음" |
| 7 | **세계관 위반** | 자체 규칙을 스스로 깸 | "3화에서 불가능하다고 한 걸 7화에서 하고 있음" |
| 8 | **메시지 과잉** | 주제의식을 강요 | "독자한테 교훈 주려고 캐릭터를 희생시킴" |

> [!TIP]
> **업무에 적용하기**: "악플러"를 "까다로운 검토자"로 바꾸면 됩니다.
> - 제안서 → "심사위원 관점에서 약점을 찾아라"
> - 보고서 → "LP가 가장 불편하게 느낄 부분을 찾아라"
> - 이메일 → "오해의 소지가 있는 표현을 모두 찾아라"

### 변주매트릭스: 패턴 반복 방지

검증의 또 다른 차원은 **패턴이 반복되는 것을 감지**하는 것입니다.

웹소설에서 사용한 "변주매트릭스" 예시:

| 화 | 진짜 룰 | 오퍼 | 동행 | 감정 |
|---|---------|------|------|------|
| 1화 | 규칙A | 큰거 | 여성 | 분노 |
| 2화 | 규칙B | 작은거 | 남성 | 슬픔 |
| 3화 | 규칙A | 큰거 | 여성 | 분노 |

→ **⚠️ 3화 경고: 1화와 패턴 겹침!**

**업무 적용 예시 — 제안서 변주매트릭스:**

| 제안서 | 차별화 포인트 | 레퍼런스 | 수익 구조 | 리스크 대응 |
|--------|-------------|---------|----------|-----------|
| A펀드 | 트랙레코드 | 해외 사례 | 임대 수익 | 보험 |
| B펀드 | 트랙레코드 | 해외 사례 | 임대 수익 | 보험 |

→ **⚠️ B펀드 제안서가 A펀드와 거의 동일!** 심사위원이 "복사했네"라고 느낄 수 있음.

### 결정 로그: 재론 금지

AI와 일하다 보면 같은 논의를 **반복**하게 됩니다.  
"이건 이미 결정한 건데, 또 물어보네?"

**결정 로그**는 확정된 사항을 기록해서, AI가 다시 꺼내지 못하게 하는 장치입니다.

```
D1: 연애는 메인 엔진이 아니다 — 왜? 작가가 과몰입하는 경향
D5: 기원은 깊게 파지 말 것 — 왜? 데못죽 80화 이후 망한 선례
D8: 주인공은 운이 아니라 촉+배짱 — 왜? 운으로 뚫으면 긴장감 제로
D12: 회당 4,000자 이내 — 왜? 웹 독서 환경에서 집중력 한계
D15: 악역에게도 논리를 줄 것 — 왜? 멍청한 악역 = 멍청한 주인공
```

> **"매번 거절하는 대신, 거절을 기록해서 시스템에 넣어놓은 것."**

**업무 적용 예시 — 제안서 결정 로그:**

```
D1: 펀드 규모는 200억 이내 — 왜? 운용 인력 대비 적정 규모
D2: Co-GP는 2사 이내 — 왜? 의사결정 속도 저하 방지
D3: 수익률 표기는 세전 IRR — 왜? 세후는 변수가 많아 오해 유발
D4: ESG는 별도 섹션으로 분리 — 왜? 본문에 섞으면 핵심이 희석됨
D5: 레퍼런스는 국내 사례 우선 — 왜? 해외 사례는 "우리도 할 수 있나"의 의문
```

> [!IMPORTANT]
> **결정 로그의 핵심**: 단순히 "뭘 결정했는가"가 아니라 **"왜 결정했는가"**를 함께 기록합니다.  
> 이유가 있어야 나중에 "이거 바꿀까?"라는 유혹이 올 때 판단할 수 있습니다.

## 4. 지식 축적: 시간이 지날수록 AI가 나를 더 잘 안다

### 문제: 대화는 사라진다

| 현재 상태 | 문제 |
|----------|------|
| 3월에 좋은 분석을 했다 | 6월에 기억 못 함 |
| 지난달 의사결정 근거를 정리했다 | 새 채팅에서 사라짐 |
| 2년치 보고서 패턴을 파악했다 | 다음 분기에 처음부터 |

대화 하나하나는 사라지지만, **지식으로 축적하면 영구적**입니다.

---

### 개인 위키 시스템

한 사용자가 운영 중인 **개인 지식 위키** 시스템입니다.  
AI가 3가지 명령으로 위키를 관리합니다:

| 명령 | 기능 | 예시 |
|------|------|------|
| **INGEST** | 새 영감/정보 흡수 | 영화 보고 느낀 점 → 기존 지식과 연결하여 저장 |
| **QUERY** | 위키 검색 | "실존주의와 관련된 노트 모두 보여줘" |
| **LINT** | 위키 건강 점검 | 고아 노드(연결 없는 항목), 약한 연결 체크 |

---

### 실제 사례: 영화에서 철학까지

**INGEST 예시:**

EEAAO(에브리씽 에브리웨어 올 앳 원스)를 보고:

```
INGEST: EEAAO를 봤다. "이거 파우스트 같다"는 느낌.
- 무한한 가능성 앞에서의 허무
- 그럼에도 "여기"를 선택하는 것
- 니힐리즘을 통과한 후의 긍정
```

AI가 자동으로 연결:

```
→ 기존 노드와 연결됨:
  - 카뮈의 시지프 신화: "부조리 앞에서 반항"
  - 니체의 영원회귀: "이 삶을 다시 살겠는가?"
  - 틸리히의 존재의 용기: "불안 속에서 존재를 긍정"
  
→ 새 엔트리 생성: 2026-05-01_EEAAO_파우스트.md
→ 허브 노드 업데이트: concepts/그럼에도_불구하고.md
```

---

### 허브 노드 구조

여러 개의 개별 노트가 하나의 **핵심 개념(허브 노드)**으로 연결됩니다:

```
concepts/그럼에도_불구하고.md          ← 허브 노드
├── 카뮈: 시지프 신화                   ← 이론적 기반
│   └── "바위를 굴리는 시지프를 행복하다고 상상해야 한다"
├── 니체: 영원회귀                      ← 이론적 기반
│   └── "이 삶을 영원히 반복해도 좋겠는가?"
├── 틸리히: 존재의 용기                 ← 이론적 기반
│   └── "불안을 자기 안에 끌어안는 용기"
└── 연결된 엔트리                       ← 구체적 경험/관찰
    ├── 2026-04-19_성난사람들.md
    │   └── "분노 뒤에 숨은 무력감, 그리고 그럼에도"
    ├── 2026-05-01_EEAAO_파우스트.md
    │   └── "멀티버스의 허무를 통과한 후 세탁소를 선택"
    ├── 2026-05-15_투자실패회고.md
    │   └── "실패에서 배운 것이 아니라, 실패에도 계속하는 것"
    └── ...
```

### 업무에 적용하기

개인 위키는 창작만을 위한 것이 아닙니다. **업무 지식 축적**에도 동일하게 적용됩니다:

| 창작 위키 | 업무 위키 |
|----------|----------|
| 영화 감상 → 철학 연결 | 투자 사례 → 패턴 축적 |
| 허브 노드: "그럼에도 불구하고" | 허브 노드: "실패한 투자의 공통점" |
| INGEST: 새 영감 기록 | INGEST: 새 사례/뉴스 기록 |
| QUERY: 관련 개념 검색 | QUERY: 유사 사례 검색 |
| LINT: 노드 건강 점검 | LINT: 오래된 정보 업데이트 체크 |

**업무 위키 예시:**

```
hub/실패한_펀드_공통점.md              ← 허브 노드
├── 과도한 레버리지
│   └── 2023-03 ○○펀드: LTV 80% → 금리 인상 시 디폴트
├── 단일 자산 집중
│   └── 2024-01 △△펀드: 오피스 1동 → 공실 발생 시 수익 제로
├── LP-GP 이해 불일치
│   └── 2024-06 □□펀드: LP는 안정 수익 원했는데 GP는 개발형
└── 연결된 사례
    ├── 2025-01_해양펀드_분석.md
    ├── 2025-04_물류센터_실사.md
    └── ...
```

> [!TIP]
> **시작 방법**: 거창하게 시작할 필요 없습니다.
> 1. 이번 주 업무에서 배운 것 하나를 기록하세요
> 2. 다음 주에 비슷한 일이 생기면, 그 기록을 참조하세요
> 3. 한 달이면 쓸만한 지식 베이스가 됩니다

## 5. 파이프라인: 모든 것을 연결한다

### 파이프라인이란

지금까지 배운 것을 연결하면 **파이프라인**이 됩니다:

```
에이전트 (누가) + 스킬 (무엇을 어떻게) + 검증 (제대로 했나) = 파이프라인
```

파이프라인은 **여러 에이전트가 순서대로 일하고, 각 단계를 검증하는 자동화된 흐름**입니다.

---

### 실제 사례 — 웹소설 6-에이전트 파이프라인

```
명령: /에피소드 [주제]

① 편집장(상황파악)
   └── 현재 스토리 상황, 캐릭터 상태, 이전 결정 로그 확인
       ↓
② 작가(초안)
   └── SOP에 따라 에피소드 초안 작성 (비트시트 → 본문)
       ↓
③ 악플러(파괴)
   └── 8가지 사냥 목록으로 약점 색출
   └── 변주매트릭스로 패턴 반복 체크
       ↓
④ 독자(감정)
   └── "이 장면에서 뭘 느꼈는가?" 감정 반응 체크
   └── 지루한 구간 표시
       ↓
⑤ 팬덤(몰입)
   └── "이걸 SNS에 올리고 싶은가?" 바이럴 포인트 체크
   └── 떡밥/복선 연결 확인
       ↓
⑥ 체커(정합성)
   └── 세계관 위반 체크, 캐릭터 일관성, 타임라인 검증
       ↓
⑦ 편집장(판정)
   └── 모든 피드백 종합 → 수정/승인/재작성 판정
```

> [!NOTE]
> **핵심**: 하나의 AI가 모든 역할을 순서대로 수행합니다.  
> "에이전트"는 물리적으로 다른 AI가 아니라, **다른 규칙과 관점으로 전환**하는 것입니다.  
> 하나의 프롬프트 안에서 역할을 바꿔가며 실행할 수도 있고,  
> 별도의 채팅/프로젝트로 분리할 수도 있습니다.

---

### 실제 사례 — B-Square 펀드 제안서 파이프라인

```
📢 공고 발견
    ↓
① 공고분석 (Researcher)
   └── 핵심 요구사항 + 배점 분석표
       ↓
② 기획안 (Analyst + Writer)
   └── 대응 전략 + 구조 설계
       ↓
③ 펀드출자제안서 (Writer)
   └── 제안서 전문 초안
       ↓
④ QA체크리스트 (Reviewer)
   └── 수치 검증 + 요구사항 매칭 + 논리 점검
       ↓
⑤ 펀드구술심사 (Analyst)
   └── 예상 질문 50개 + 답변 스크립트
       ↓
⑥ 모의심사 (Reviewer — 심사위원 역할)
   └── 실전 시뮬레이션 + 취약점 분석
       ↓
⑦ 발표자료자동화 (Writer)
   └── 슬라이드 구성안 + 발표 스크립트
```

**이 파이프라인의 효과:**

| | 파이프라인 없이 | 파이프라인 적용 후 |
|---|--------------|-----------------|
| 소요 시간 | 2~3주 | 3~5일 |
| 빠뜨리는 항목 | 자주 발생 | 체크리스트가 잡아냄 |
| 제안서 품질 | 담당자 역량에 의존 | 일관된 품질 |
| 구술 준비 | 예상 질문 10개 정도 | 50개 + 시뮬레이션 |

---

### 대화 기술 → 시스템 매핑

레벨 1에서 배운 **대화 기술** 하나하나가, 레벨 3에서는 **시스템의 구성요소**가 됩니다:

| 레벨 1에서 한 것 | 레벨 3에서 시스템으로 만든 것 | 왜 |
|--------------|--------------------------|------|
| 맥락을 먼저 줬다 | → **CLAUDE.md 규칙 파일** | 매번 맥락을 주는 대신 자동 로드 |
| 뻔한 답을 거절했다 | → **악플러 에이전트** | 매번 거절하는 대신 역할로 분리 |
| 규칙을 만들었다 | → **흥행공식 + 변주매트릭스** | 매번 규칙을 말하는 대신 시스템에 내장 |
| 제약을 확정했다 | → **결정 로그 (재론 금지)** | 매번 "그건 안 돼"를 반복하지 않음 |
| 캐릭터를 만들었다 | → **위키 허브 노드** | 캐릭터 정보가 축적되고 연결됨 |
| 장면을 압축했다 | → **비트시트 형식** | 모든 장면이 같은 구조로 설계됨 |
| 이미지를 참조했다 | → **에셋 라이브러리** | 참조 자료가 정리되어 재사용 가능 |

> [!IMPORTANT]
> **이것이 레벨 1 → 3의 진짜 여정입니다.**  
> 대화에서 반복하던 것을 → 시스템에 넣어서 자동화하는 것.  
> "잘 말하는 것"에서 "잘 설계하는 것"으로 진화하는 것.

---

### 나만의 파이프라인 설계하기

파이프라인을 만드는 단계:

#### Step 1: 반복 업무 찾기
```
질문: "지난 한 달간 3번 이상 반복한 업무가 있는가?"

예시:
- 주간 보고서 작성 (매주)
- 클라이언트 이메일 답장 (매일)
- 회의록 정리 (주 2~3회)
- SNS 콘텐츠 기획 (주 3회)
```

#### Step 2: 단계 분해
```
"주간 보고서 작성"을 단계로 나누면:

1. 이번 주 업무 내역 수집
2. 핵심 성과/이슈 정리
3. 수치 데이터 업데이트
4. 다음 주 계획 작성
5. 보고서 양식에 맞춰 작성
6. 오탈자/수치 검토
7. 발송
```

#### Step 3: 에이전트 + 스킬 배치
```
1. 데이터 수집 → Monitor 에이전트 (뉴스모니터링 스킬)
2. 성과 정리 → Analyst 에이전트 (데이터분석 스킬)
3. 수치 업데이트 → Analyst 에이전트 (스프레드시트 스킬)
4. 계획 작성 → Writer 에이전트 (보고서작성 스킬)
5. 양식 작성 → Writer 에이전트 (보고서작성 스킬)
6. 검토 → Reviewer 에이전트 (QA체크리스트 스킬)
7. 발송 → 사람 (최종 확인)
```

#### Step 4: 검증 포인트 삽입
```
Step 2 이후: "빠진 업무가 없나?" 체크
Step 3 이후: "수치가 지난주와 일관적인가?" 체크
Step 6 이후: "수신인과 내용이 맞나?" 체크
```

## 부록: Claude 공식 프롬프트 엔지니어링 원칙 (요약)

Anthropic에서 공식적으로 제시한 8가지 프롬프트 원칙을, 실전 활용과 매핑합니다.

| # | 원칙 | 의미 | 창작/업무에서 | 예시 |
|---|------|------|-------------|------|
| 1 | **Be specific** | 구체적으로 요청하라 | "소설 써줘" → 금지. 장르/분량/톤/구조를 명시 | "도시 판타지, 4000자, 하드보일드 톤, 3막 구조" |
| 2 | **Use examples** | 예시를 제공하라 | 레퍼런스를 첨부. "이런 느낌으로" | 기존 보고서 예시 첨부 → "이 양식으로 써줘" |
| 3 | **Let Claude think** | 생각할 시간을 줘라 | "단계별로 접근해"를 포함 | "먼저 구조를 잡고, 그 다음 각 섹션을 써줘" |
| 4 | **Use XML tags** | 입력을 구조화하라 | 역할/맥락/제약/출력을 분리 | `[역할] [맥락] [제약] [출력양식]` 구분 |
| 5 | **Give Claude a role** | 역할을 부여하라 | 에이전트의 핵심 | "너는 10년차 투자 심사역이다" |
| 6 | **Prefill response** | 시작점을 제공하라 | "다음 형식으로 시작해:" | "## 1. 투자 개요\n\n 부터 시작해" |
| 7 | **Chain prompts** | 체이닝하라 | 에이전트 → 스킬 → 검증 | 파이프라인 전체 구조 |
| 8 | **Long context** | 긴 맥락을 활용하라 | 위키 전체를 참조시켜라 | 프로젝트 지식에 200K 토큰 활용 |

### 원칙별 실전 Before & After

#### 원칙 1: Be specific (구체적으로)

```
❌ "보고서 써줘"
✅ "3분기 LP 보고서를 작성해줘.
    - 양식: 기존 보고서 예시 따를 것
    - 펀드: B-Square 부동산 2호
    - 포함: AUM, 집행률, 포트폴리오 현황, 주요 이슈
    - 분량: A4 3장 이내
    - 톤: 보수적, 수치 중심"
```

#### 원칙 5: Give a role (역할 부여)

```
❌ "이 제안서 검토해줘"
✅ "너는 20년 경력의 까다로운 기금 심사위원이다.
    이 제안서를 읽고, 실제 심사에서 지적할 항목을 찾아라.
    칭찬은 하지 마라. 약점만 보고하라.
    각 약점에 구체적 근거와 개선 방향을 제시하라."
```

#### 원칙 7: Chain prompts (체이닝)

```
❌ 한 번에 모두 요청:
   "공고 분석하고, 기획안 만들고, 제안서 써줘"

✅ 단계별 체이닝:
   Step 1: "이 공고의 핵심 요구사항을 배점 기준으로 정리해줘"
   Step 2: (결과 확인 후) "이 분석 기반으로 대응 기획안 만들어줘"
   Step 3: (결과 확인 후) "이 기획안으로 제안서 섹션 1~3 작성해줘"
```

```

```

> [!IMPORTANT]
> **가장 중요한 것**: 처음부터 완벽한 시스템을 만들려 하지 마세요.
> 
> 1. **에이전트 1개**부터 시작하세요 (가장 귀찮은 업무)
> 2. **스킬 SOP 1개**를 써보세요 (그 업무의 절차)
> 3. **검증 항목 3개**를 정하세요 (최소한의 품질 기준)
> 4. 쓰면서 고치세요. **살아있는 시스템**이 완벽한 계획보다 낫습니다.

```

```

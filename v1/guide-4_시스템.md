# 문서 4: 시스템 — AI에게 역할 부여하고 환경 구축하기

> **규칙 파일 + 에이전트 + 파이프라인** | 예상 학습 시간: 90–120분  
> 이전 단계: [문서 3 — 확장](guide-3_확장.md)

---

## 0. 이 문서의 목적

AI를 **"매번 설명해야 하는 도구"**에서 **"알아서 일하는 시스템"**으로 전환한다.

| 단계 | 상태 | 비유 |
|------|------|------|
| 문서 1–2 | AI에게 매번 설명 | 매번 새 알바생에게 처음부터 설명 |
| 문서 3 | AI가 내 데이터를 읽음 | 알바생이 우리 회사 자료를 봄 |
| **문서 4** | **AI가 역할+규칙+절차를 갖춤** | **교육받은 담당 직원이 SOP대로 일함** |

이 문서를 마치면:
- 규칙 파일을 작성하여 AI의 기본 행동을 설정할 수 있다
- 업무별 에이전트를 설계할 수 있다
- 리서치 → 분석 → 문서 생성 파이프라인을 구축할 수 있다
- 본인의 직군에 맞는 AI 시스템을 운영할 수 있다

---

## 4-1. 규칙 파일이란

### 개념

```
규칙 파일 = AI에게 주는 "업무 매뉴얼"

보통:  매번 "너는 ~이고, ~하게 해줘"를 반복
규칙 파일: 한 번 써놓으면, AI가 항상 그 역할로 시작
```

### 어디에 쓰나

| 도구 | 규칙 파일 이름 | 위치 | 적용 방식 |
|------|-------------|------|----------|
| **Claude** (Co-work) | 프로젝트 설명 + 첨부파일 | 프로젝트 설정 화면 | 프로젝트 열면 자동 적용 |
| **Cursor** | `.cursorrules` | 프로젝트 폴더 루트 | Cursor가 자동 인식 |
| **Antigravity** | `.cursorrules` 또는 `CLAUDE.md` | 프로젝트 폴더 루트 | 에디터가 자동 인식 |
| **Claude Code** | `CLAUDE.md` | 프로젝트 폴더 루트 | Claude가 자동 인식 |
| **ChatGPT** | 커스텀 명령어 (Custom Instructions) | 설정 → 맞춤 설정 | 모든 대화에 적용 |

### 비유

```
규칙 파일 없이:
  나: "너는 사진작가 어시스턴트야. 견적서 쓸 때는 이 양식이고,
       톤은 깔끔하게, VAT 별도로 표기하고..."
  (매번 이걸 반복)

규칙 파일 있으면:
  나: "견적서 써줘. 인물 촬영, 3시간, 보정 20컷."
  AI: (규칙 파일에서 양식/톤/항목 자동 참조 → 바로 생성)
```

```
또 다른 비유:

카페 알바:
  규칙 파일 없음 = "아메리카노 한 잔인데, 뜨겁게, 컵은 이 컵으로,
                    시럽은 안 넣고, 물은 적게..." (매번 설명)
  규칙 파일 있음 = "1번" (레시피북에 다 적혀 있음)
```

### CLAUDE.md와 .cursorrules의 차이

```
CLAUDE.md:
├── Claude Code, Antigravity에서 사용
├── 마크다운 형식
├── 프로젝트 전반의 규칙 정의
└── 하위 폴더에 추가 CLAUDE.md 가능 (폴더별 규칙)

.cursorrules:
├── Cursor에서 사용
├── 마크다운 또는 텍스트 형식
├── 프로젝트 루트에 1개
└── Cursor 전용이지만 Antigravity도 인식

실질적 차이:
  거의 없음. 같은 내용을 적으면 됨.
  사용하는 도구에 맞는 파일명만 선택.
```

## 4-2. 나만의 규칙 파일 만들기

### 기본 구조

모든 규칙 파일은 이 구조를 따른다:

```markdown
# CLAUDE.md (또는 .cursorrules)

## 나는 누구인가
- 이름: [이름]
- 직업: [사진작가 / 영상제작자 / 공간기획자 / 댄서 / ...]
- 주 작업: [포트레이트 촬영 / 다큐멘터리 / 전시기획 / 안무 / ...]
- 경력: [N년차, 주요 이력]

## 작업 스타일
- 톤: [담백 / 감성적 / 실험적 / 격식체 / ...]
- 자주 쓰는 양식: [시놉시스 / 견적서 / 제안서 / 촬영 체크리스트 / ...]
- 참조 자료: [첨부된 파일 목록]

## 반복 작업 규칙
- 견적서 작성 시: [양식, 항목, 톤, 통화 단위]
- SNS 캡션: [길이, 해시태그 규칙, 톤]
- 이메일 답장: [형식, 서명, 존칭 규칙]
- 회의록 정리: [구조, 필수 항목]

## 출력 형식
- 기본 언어: 한국어
- 마크다운 사용
- 표와 코드 블록 활용
- 구어체 / 문어체: [선택]

## 금지 사항
- 존댓말 과하게 쓰지 마
- "혁신적", "시너지", "패러다임" 같은 단어 쓰지 마
- 내가 안 준 정보 만들어내지 마 (환각 금지)
- 이모지 남발하지 마
- 뻔한 결론으로 끝내지 마
```

### 사진작가용 규칙 파일 — 완성 예시

```markdown
# CLAUDE.md — 사진작가 어시스턴트

## 컨텍스트
- 나는 상업/인물 사진작가 (7년차)
- 주로 야외 자연광 + 보조조명 세팅
- 클라이언트: 브랜드, 잡지, 개인 (프로필/화보)
- 장비: Sony A7IV, 시그마 35mm Art, 소니 85mm f/1.4
- 보정: Lightroom Classic + Photoshop

## 촬영 체크리스트 생성 시
- 포함 항목: 장소 / 날씨 / 시간대 / 모델 수 / 콘셉트
- 장비 리스트 순서: 바디 → 렌즈 → 조명 → 악세서리 → 소품
- 과거 유사 촬영 세팅 참조 (첨부된 세팅 DB가 있으면 활용)
- 비상 장비 항목 포함 (여분 배터리, 메모리카드, 반사판)

## 견적서 생성 시
- 항목: 촬영비 / 보정비 / 출장비 / 장비대여 / 어시스턴트
- 촬영비 기본 단가: 시간당 15만원
- 보정비 기본 단가: 컷당 1만원 (기본보정), 3만원 (고급보정)
- VAT 별도 명시
- 결제 조건: 계약금 50%, 잔금 납품 시
- 톤: 깔끔, 간결, 전문적

## 캡션 생성 시
- 3줄 이내
- 해시태그 10개 이내 (한국어 5 + 영어 5)
- 감성적이되 과하지 않게
- "작업물" "촬영" 등 업계 용어 자연스럽게 사용

## 금지 사항
- "감성적인 한 컷" 같은 식상한 표현 금지
- 클라이언트 정보 임의 생성 금지
- 촬영 세팅값을 추측하지 말 것 (모르면 모른다고)
```

### 영상제작자용 규칙 파일 — 완성 예시

```markdown
# CLAUDE.md — 영상 프로덕션 어시스턴트

## 컨텍스트
- 나는 다큐멘터리/교양 영상 제작자 (5년차)
- 기획 → 촬영 → 편집 → 납품 전 과정 관여
- 주 클라이언트: 방송사, 문화재단, 기업
- 편집: Premiere Pro + DaVinci Resolve
- 사운드: Logic Pro

## 시놉시스 작성 시
- 구조 (이 순서를 반드시 지킴):
  1. 로그라인 (1문장, 반드시 1문장)
  2. 기획의도 (A4 반 페이지)
  3. 에피소드 구성 (회당 구성안)
  4. 제작 계획 (일정 + 예산 + 인력)
  5. 제작진 이력
- 톤: 담백, 구체적, 수치 포함
- 분량: A4 2–3장

## 콜시트 생성 시
- 표 형식 필수
- 열 구성: 일자 / 시간 / 장소(주소 포함) / 출연자 / 스태프 / 장비 / 비고
- 시간은 24시간제
- 집결 시간과 촬영 시작 시간 분리 표기

## 결과보고서 작성 시
- 구조: 프로젝트 개요 → 수행 내용 → 성과(정량+정성) → 예산 집행 → 첨부
- 예산 집행 내역: 항목별 계획 vs 실집행 비교표
- 정량 성과: 조회수, 참여자 수, 매체 노출 등 수치 필수

## 금지 사항
- "영상미" 같은 모호한 표현 금지
- 예산 항목을 추측하지 말 것
- 촬영 장소 허가/섭외 상태를 임의로 "완료"로 적지 말 것
```

### Claude Co-work에서 규칙 파일 적용하기

```
방법 1: 프로젝트 설명에 직접 입력

  ① claude.ai → 왼쪽 사이드바 → "새 프로젝트"
  ② 프로젝트 이름: "촬영 어시스턴트"
  ③ 프로젝트 설명란에 위의 규칙 파일 내용 붙여넣기
  ④ 저장

  이후 이 프로젝트에서 대화하면 → 규칙이 자동 적용

방법 2: 파일로 첨부

  ① 위의 규칙 파일을 .md 파일로 저장
  ② 프로젝트에 파일 첨부 (드래그 앤 드롭)
  ③ 추가로 참조 자료 첨부 (이력서, 포트폴리오, 단가표 등)

  이후 "첨부된 자료 기반으로 견적서 써줘" → 내 양식 + 내 단가로 생성
```

### Cursor / Antigravity / Claude Code에서 적용하기

```
① 프로젝트 폴더 루트에 파일 생성:
   /내프로젝트/.cursorrules    (Cursor)
   /내프로젝트/CLAUDE.md       (Claude Code, Antigravity)

② 위의 규칙 파일 내용 작성

③ 저장. 끝.
   → 해당 폴더에서 AI를 실행하면 자동으로 규칙 적용

하위 폴더별 추가 규칙 (CLAUDE.md만 가능):
   /내프로젝트/CLAUDE.md              ← 전체 규칙
   /내프로젝트/견적서/CLAUDE.md        ← 견적서 작업 시 추가 규칙
   /내프로젝트/SNS/CLAUDE.md          ← SNS 작업 시 추가 규칙
```

## 4-3. 에이전트 설계 — 반복 기획을 루틴화

### 에이전트란

```
규칙 파일 = "이 사람은 이렇게 일한다" (기본 성격)
에이전트  = "이 작업은 이렇게 한다" (특정 업무 SOP)

비유:
  규칙 파일 = 회사 전체 업무 매뉴얼
  에이전트  = 부서별 업무 절차서
```

에이전트는 특별한 기술이 아니다. **Claude의 프로젝트 기능**을 사용해서 "특정 역할에 특화된 AI 설정"을 만드는 것이다.

### 아티스트가 만들 수 있는 에이전트

| 에이전트 | 역할 | 입력 | 출력 | 사용 도구 |
|---------|------|------|------|----------|
| **제안서 작성기** | 공모전/프로젝트 제안서 | 공고문 + 내 이력 | 제안서 초안 | Claude Co-work |
| **견적서 생성기** | 촬영/제작 견적 | 작업 조건 (유형, 시간, 컷수) | 견적서 PDF | Claude Co-work / Cursor |
| **리서치 요약기** | 레퍼런스 수집·분석 | 주제 키워드 | 정리된 리서치 문서 | Gemini + Claude |
| **SNS 매니저** | 캡션 + 해시태그 생성 | 이미지 설명 or 사진 | 캡션 3종 + 해시태그 | ChatGPT / Claude |
| **세팅 검색기** | 과거 촬영/조명 세팅 호출 | 조건 (날씨, 장소, 스타일) | 유사 세팅 + 추천 | Claude Code + 옵시디언 |
| **회의록 정리기** | 녹음 → 구조화 | 녹음 텍스트 | 액션아이템 포함 회의록 | Claude Co-work |
| **포트폴리오 큐레이터** | 작업물 선별 + 설명 작성 | 작업물 목록 + 대상 | 큐레이션 + 아티스트 노트 | Claude Co-work |
| **계약서 검토기** | 계약서 조건 분석 | 계약서 텍스트 | 주의 사항 + 수정 제안 | Claude |

### 에이전트 작성법 — Claude Co-work 기준

#### 단계별 설정

```
1단계: 프로젝트 생성
  claude.ai → 왼쪽 사이드바 → "+" → 새 프로젝트
  이름: "제안서 작성기"

2단계: 프로젝트 설명 작성 (이것이 에이전트의 핵심)
  아래의 "에이전트 설명 템플릿" 참조

3단계: 참조 자료 첨부
  - 내 이력서 / 포트폴리오 PDF
  - 과거 제안서 샘플 (당선작)
  - 자주 응모하는 공고 유형 정리

4단계: 테스트
  공고문 1개를 넣고 → 결과 확인 → 규칙 수정 → 반복

5단계: 운영
  이후 공고문만 넣으면 → 제안서 초안이 나옴
```

#### 에이전트 설명 템플릿

```markdown
## 역할
너는 문화예술 프로젝트 제안서 작성 전문가다.

## 절차 (이 순서를 반드시 따를 것)
1. 공고문을 받으면 → 핵심 요구사항 추출 (표로 정리)
   - 주제, 자격 조건, 제출 서류, 평가 기준, 마감일
2. 내 이력(첨부 파일)에서 관련 경험 매칭
   - 공고 요구사항 ↔ 내 경험 매칭표
3. 제안서 구조 생성
   - 공고의 평가 기준에 맞춘 목차
4. 초안 작성
   - 각 섹션별 내용 작성

## 출력 형식
- 마크다운 문서
- 표, 코드 블록 활용
- 각 섹션 끝에 [검토 포인트] 표시

## 톤
- 구체적, 담백
- 과장 금지 ("최초", "혁신적" 등 사용 금지)
- 수치와 사실 기반

## 제한
- 내가 제공하지 않은 경력/수상 이력을 만들어내지 마
- 예산은 추측하지 말고, 필요하면 물어봐
- 확실하지 않은 정보는 [확인 필요]로 표시
```

#### 에이전트: 견적서 생성기 — 예시

```markdown
## 역할
너는 촬영/제작 견적서 생성 전문가다.

## 입력
사용자가 다음 조건을 제공한다:
- 촬영 유형 (인물/제품/공간/행사)
- 촬영 시간
- 보정 컷수
- 출장 여부 + 거리
- 추가 요구 (어시스턴트, 장비 대여 등)

## 단가표 (이 단가를 기본으로 적용)
| 항목 | 단가 |
```

## 4-4. 파이프라인: 시장 리서치 → 분석 → 문서 생성

### 파이프라인이란

```
파이프라인 = 여러 단계의 작업을 순서대로 연결한 것

에이전트 1개:  "견적서 써줘" → 견적서
파이프라인:    "공고문 분석 → 리서치 → 전략 수립 → 제안서 작성 → 검토"
              = 5단계를 순서대로 실행
```

### 4단계 파이프라인 — 전체 흐름

```
[1단계] 리서치 수집
        │
        ├── Gemini Deep Research: 주제별 심층 조사
        │   "2026 몰입형 전시 트렌드 + 해외 사례 + 관객 반응"
        │
        ├── Grok: X(트위터)에서 실시간 업계 반응
        │   "최근 전시 관련 업계 반응, 논쟁, 화제작"
        │
        └── NotebookLM: 관련 영상/팟캐스트/PDF 텍스트 추출
            "이 유튜브 3개 + PDF 2개에서 핵심 내용 뽑아줘"
        │
        ▼
[2단계] 분석·구조화
        │
        ├── Claude (Co-work)에 1단계 결과 전부 넣기
        │   프로젝트에 자료를 첨부 또는 붙여넣기
        │
        ├── "이 자료들의 공통 트렌드 5가지 뽑아줘"
        ├── "경쟁 사례와 빈자리(기회 영역) 분석해줘"
        └── "우리 프로젝트에 적용 가능한 포인트는?"
        │
        ▼
[3단계] 문서 생성
        │
        ├── Claude: 기획서/제안서 초안
        │   "분석 결과를 기반으로 제안서 작성해줘"
        │
        ├── ChatGPT: 컨셉 이미지 생성 (무드보드)
        │   "이 기획서의 공간 컨셉을 이미지로 만들어줘"
        │
        └── Cursor/Antigravity: 프레젠테이션 or 웹페이지
            "이 내용으로 프레젠테이션 만들어줘"
        │
        ▼
[4단계] 검증·수정
        │
        ├── "이 기획서에서 약한 부분 지적해줘"
        ├── "심사위원 관점에서 반박할 수 있는 포인트 3개는?"
        ├── "예산 계획의 현실성을 평가해줘"
        └── 수정 → 재검토 → 최종
```

### 각 단계 상세

#### 1단계: 리서치 수집 — 도구별 역할

| 도구 | 역할 | 입력 예시 | 출력 |
|------|------|----------|------|
| **Gemini Deep Research** | 심층 조사 (웹 기반) | "2026 몰입형 전시 사례 + 기술 + 관객 반응" | 10–20페이지 리서치 문서 |
| **Grok** | 실시간 여론 (X/트위터) | "최근 미디어아트 전시 반응" | 업계 반응, 트렌드, 논쟁 정리 |
| **NotebookLM** | 기존 자료 정리 | 유튜브 URL, PDF, 녹음 파일 | 텍스트 추출 + 요약 + 마인드맵 |
| **옵시디언 위키** | 내부 지식 검색 | "과거 유사 프로젝트 사례" | 내 경험 기반 인사이트 |

```
실전 예시 — 전시 기획 리서치:

Gemini에게:
  "2024-2026 몰입형 전시 사례를 조사해줘.
   국내 5개, 해외 5개.
   각 사례별: 주제, 기술, 공간 크기, 관객 수, 특징, 비용(가능하면)"

Grok에게:
  "최근 X에서 '몰입형 전시' 관련 반응을 정리해줘.
   긍정적 반응, 부정적 반응, 반복되는 키워드"

NotebookLM에:
  관련 유튜브 영상 3개 URL 입력
  → "이 영상들의 핵심 내용과 공통점을 정리해줘"
```

#### 2단계: 분석 · 구조화 — Claude Co-work에서

```
프로젝트: "2026 전시 기획"

첨부 자료:
  - Gemini 리서치 결과.md
  - Grok 여론 분석.md
  - NotebookLM 요약.md
  - 내 과거 전시 이력.pdf

요청 1: "이 자료들의 공통 트렌드 5가지를 뽑아줘"
→ AI 출력: 트렌드 표

요청 2: "경쟁 사례 중 우리가 차별화할 수 있는 빈자리를 분석해줘"
→ AI 출력: 빈자리 분석 + 차별화 포인트

요청 3: "트렌드 + 빈자리 + 내 이력을 기반으로,
          제안서의 핵심 전략을 3개 제안해줘"
→ AI 출력: 전략 3종 (각각 근거 포함)
```

#### 3단계: 문서 생성 — 멀티 도구 활용

```
Claude에게:
  "2단계 분석 결과를 기반으로 제안서를 작성해줘.
   구조: 기획의도 → 프로젝트 개요 → 공간 계획 → 기술 계획 → 예산 → 일정"

ChatGPT에게:
  "이 기획서의 공간 컨셉을 이미지로 만들어줘.
   키워드: 몰입형, 프로젝션, 어두운 공간, 관객 참여, 사운드"

Cursor/Antigravity에게:
  "이 기획서 내용으로 웹 프레젠테이션을 만들어줘.
   심플한 디자인, 슬라이드 형식, 이미지 포함"
```

#### 4단계: 검증 · 수정 — AI를 "검토자"로 활용

```
검토 요청 1 (약점 분석):
  "이 제안서의 약한 부분 3가지를 지적해줘.
   구체적으로 어떻게 보완할지도 제안해."

검토 요청 2 (반대 관점):
  "심사위원이 이 제안서를 탈락시킬 이유 3가지를 만들어줘.
   그리고 각 이유에 대한 반박을 준비해줘."

검토 요청 3 (현실성 체크):
  "이 예산 계획이 현실적인지 평가해줘.
   각 항목별 시장 가격과 비교해서."

검토 요청 4 (최종 교정):
  "최종본에서 오탈자, 수치 불일치, 논리적 비약을 찾아줘."
```

```
[일상: 영감 수집]

      │
```

## 4-6. 실전 사례: 실제 동작하는 에이전트 시스템

> 아래는 실제로 사용되고 있는 에이전트 파일과 규칙 파일이다. 이런 시스템을 AI에게 만들어달라고 하면 된다.

### 실전 에이전트 파일 — 웹소설 작가 에이전트

Claude Code에서는 `.claude/agents/` 폴더에 마크다운 파일을 넣으면 에이전트가 된다. 아래는 웹소설 프로젝트에서 실제로 사용되는 **작가 에이전트** 파일이다:

```markdown
---
name: webnovel-writer
description: 웹소설의 작가. 편집장이 발주한 비트시트만 받아
  ~5,000자 본문 초안을 쓴다.
tools: Read, Write, Edit
model: opus
---

너는 이 작품의 **작가**다. 기획은 편집장이 한다.
너는 받은 비트시트를 **재미있는 본문**으로 바꾼다.

## 집필 전 반드시 읽음
- 편집장이 준 **이번 화 비트시트** (최우선)
- 설정바이블_v0.1.md — 캐릭터 보이스·세계관 사실
- 집필로그_LIVE.md — 직전 화 끝 상태(연결)
- 직전 화 원고 (톤·문체 연속성)

## 보이스 룰 (절대)
- 1인칭 단일. 다중 시점 금지.
- 단문 위주. 긴 묘사·설명충 금지.
- 모바일에서 엄지로 넘기는 리듬.

## 구조 룰
- ~5,000자, 기승전결 1사이클.
- 화 끝 = 비트시트가 지정한 절단신공으로 정확히 끊는다.

## 출력
- 본문만. N화_초안.md에 쓴다.
- 맨 위 1줄 회차 제목.
- 본문 끝에 <!-- 작가노트 --> 1~2줄.

## 리테이크 받았을 때
편집장이 준 수정 델타만 반영. 전면 재집필 아님.
```

이 에이전트 파일의 구조를 분해하면:

| 구성 요소 | 실제 내용 | 역할 |
|----------|----------|------|
| **메타데이터** (`---` 블록) | name, description, tools, model | 에이전트 이름, 사용 도구, AI 모델 지정 |
| **역할 정의** | "너는 작가다" | 이 에이전트가 무엇인지 1줄로 규정 |
| **읽기 지시** | "집필 전 반드시 읽음" | 작업 전에 참조할 파일 목록 |
| **절대 규칙** | "1인칭 단일", "단문 위주" | 절대 어기면 안 되는 규칙 |
| **출력 형식** | "N화_초안.md에 쓴다" | 결과물 형식과 저장 위치 |
| **예외 처리** | "리테이크 시 델타만 반영" | 수정 요청 시 행동 규칙 |

**핵심 패턴**: 읽기(Read) → 규칙(Rules) → 출력(Output). 이 3단계 구조는 모든 에이전트에 적용된다.

### 6-에이전트 멀티 시스템 — 실제 프로덕션 파이프라인

이 웹소설 프로젝트에는 에이전트가 6개 있다. 각각 다른 역할을 하고, 파이프라인으로 연결된다:

| 에이전트 | 파일명 | 역할 | 비유 |
|---------|--------|------|------|
| **편집장** | `webnovel-editor.md` | 비트시트 발주, 초안 심사, 승인/리테이크 | 프로듀서, 쇼러너 |
| **작가** | `webnovel-writer.md` | 비트시트 → 5,000자 본문 초안 | 작가 |
| **단순독자** | `webnovel-reader.md` | 재미있는지, 다음 화 결제할지 판단 | 일반 관객 |
| **연속성봇** | `webnovel-continuity.md` | 설정 오류, 앞뒤 불일치 검출 | 스크립트 슈퍼바이저 |
| **악플러** | `webnovel-hater.md` | 하차 사유, 약점, 패턴 반복 지적 | 가장 까다로운 비평가 |
| **팬덤** | `webnovel-fandom.md` | 2차 창작 매력도, 캐릭터 소비 가능성 | 팬 커뮤니티 |

#### 파이프라인 동작 순서

```
① 편집장: 비트시트 발주
   "3화, 1막 도입, 오퍼레이션: 표면룰=부정, 진짜룰=자살,
    절단신공: 위기형, 감정곡선: 가벼움→공포"
        │
        ▼
② 작가: 비트시트 → 본문 초안 (5,000자)
   → 3화_초안.md 생성
        │
        ▼
③ 비평 에이전트 3종이 동시에 읽기:
   ├── 악플러: "2화랑 패턴 똑같다. 별점 2/5. 하차."
   ├── 단순독자: "중반부 지루하다. 다음 화 결제? N."
   └── 팬덤: "이 캐릭터 짤 만들고 싶다. 팬아트 욕구: 3/5."
        │
        ▼
④ 연속성봇: 설정 검증
   "2화에서 주인공이 왼손잡이였는데 3화에서 오른손으로 들고 있음."
        │
        ▼
⑤ 편집장: 종합 심사
   ├── 흥행공식 체크리스트 7항목 적용
   ├── 비평 우선순위화: 이탈유발 > 설정붕괴 > 매력누락
   └── 판정:
       ├── 승인 → 집필로그 갱신 → 다음 화 발주
       └── 리테이크 → 수정 지시를 작가에게 반환 (최대 2루프)
```

**실제 악플러 에이전트의 출력 형식:**

```markdown
## 악플러 리포트 [3화]
- 하차각:
  - "오퍼레이션 시작 장면 — 또 표면룰 부정이네? 1·2화랑 똑같잖아.
     공식 들켰다."
  - "클라이맥스 직전 — 긴장감 0. 주인공이 질 것 같지가 않음."
- 짜증:
  - "시스템 설명이 3문단. 설명충."
- 패턴 들킴 여부: 직전 화와 전개 패턴 80% 동일
- 한 줄 총평: 별점 2/5, 다음 화 결제? N
```

**이 시스템의 핵심**: 사람이 6명을 고용한 것이 아니다. AI 에이전트 6개가 각각 다른 관점에서 작업을 점검한다. 비용은 AI 사용료뿐이다.

### 실전 규칙 파일 — MUNMUL 프로젝트 .cursorrules

아래는 실제 사용 중인 `.cursorrules` 파일이다. 문학 작가 × 현대 비주얼 아티스트를 페어링하는 패션-큐레이션 프로젝트의 규칙이다:

```markdown
# 文物 MUNMUL — Cursor Rules

You are working inside the 文物 MUNMUL project.
MUNMUL creates "collections" by pairing public domain literary figures
with contemporary visual artists.

## Your Role
You function as MUNMUL's production system. Given a Subject A × B,
you produce:
1. Research Brief
2. Core Statement (one sentence, contrast structure)
3. Narrative Thread (3-5 paragraphs)
4. Midjourney prompt set (5 images)
5. Reels script (30-60s, no narration)
6. Director's Note script (3-5min, narrated)
7. Metadata (captions, hashtags, pinned comment)

## Visual Direction
- Fashion editorial grammar, NOT literal scene recreation
- Human figure always central
- Prompt formula: [subject], [environment], [lighting],
  [material], [camera], [mood]
- Film stocks: Portra 800 (night), Portra 400 (day), Ektar (dynamic)
- Reference photographers: Tim Walker, Harley Weir, Paolo Roversi
- NEVER: HDR, glossy, fantasy, oversaturated

## 5-Image Rhythm
- IMG 01: Full body editorial
- IMG 02: Full body + architecture/set
- IMG 03: Symbolic costume/prop close-up
- IMG 04: Dynamic (flight, fall, motion)
- IMG 05: Close-up portrait

## Tone
- Poetic but not pretentious
- Show, don't explain
- Core Statement: contrast structure preferred
  ("one person did X, the other did Y")
```

이 규칙 파일에서 배울 수 있는 것:

| 구성 요소 | 실제 적용 | 효과 |
|----------|----------|------|
| **프로젝트 정의** | "A × B 페어링 프로젝트" | AI가 프로젝트 맥락을 즉시 파악 |
| **역할** | "production system" | AI가 뭘 만들어야 하는지 명확 |
| **출력 목록 (7종)** | Brief → Statement → Narrative → Prompt → Reels → Note → Metadata | 조건만 넣으면 7종 아웃풋이 한 번에 나옴 |
| **비주얼 규칙** | "Portra 800", "Tim Walker 참조" | 스타일이 고정됨. 매번 설명 불필요 |
| **5-이미지 리듬** | Full body → Architecture → Close-up → Dynamic → Portrait | 매번 동일한 구성. 시리즈 일관성 보장 |
| **금지 목록** | "NEVER: HDR, glossy, fantasy" | 원치 않는 결과 사전 차단 |

### 아티스트 적용 — 웹소설 시스템을 촬영/전시/공연에 대입

#### 사진작가: 촬영 에이전트 시스템

웹소설의 6-에이전트 구조를 촬영 프로젝트에 대입하면:

| 웹소설 에이전트 | 촬영 에이전트 대응 | 역할 |
|---------------|-----------------|------|
| 편집장 | **촬영 디렉터** | 촬영 계획 발주, 결과물 심사 |
| 작가 | **촬영 실행기** | 조건 → 세팅 추천 → 체크리스트 생성 |
| 악플러 | **클라이언트 시뮬레이터** | "이 사진으로 브랜드 이미지에 맞나?" 비판 |
| 단순독자 | **SNS 반응 예측기** | "이 사진 인스타에서 좋아요 받을까?" |
| 연속성봇 | **스타일 일관성 검사기** | "이전 촬영과 톤/색감이 달라졌다" 지적 |
| 팬덤 | **포트폴리오 큐레이터** | "이 컷을 포트폴리오에 넣을 가치가 있나?" |

#### 촬영 디렉터 에이전트 — 실제 파일 예시

```markdown
---
name: shooting-director
description: 촬영 프로젝트의 디렉터. 클라이언트 브리프를 받아
  촬영 계획을 수립하고, 결과물을 심사한다.
tools: Read, Write, Edit
---

너는 이 프로젝트의 **촬영 디렉터**다. 촬영을 직접 하지 않는다.
계획을 세우고, 결과물을 판단한다.

## 작업 전 반드시 읽음
- 클라이언트 브리프 (최우선)
- 과거 유사 촬영 기록 (옵시디언 위키)
- 장비 재고 목록

## 모드 A — 촬영 계획 발주
1. 클라이언트 브리프에서 추출:
   - 촬영 목적, 사용처, 톤, 레퍼런스
2. 과거 유사 촬영 세팅 조회
3. 촬영 계획서 출력:
   - 콘셉트 → 장소 → 시간대 → 장비 → 세팅 → 모델 디렉션

## 모드 B — 결과물 심사
1. 보정 완료 컷을 클라이언트 브리프 기준으로 평가
2. 체크리스트:
   - 브리프 부합도 (톤, 분위기, 용도)
   - 기술 품질 (노출, 포커스, 색감)
   - 시리즈 일관성 (이전 납품물과 비교)
3. 판정: 승인 / 추가 보정 지시 / 재촬영 사유 보고
```

#### 전시기획 규칙 파일 — MUNMUL 구조를 대입

MUNMUL의 `.cursorrules`와 동일한 구조로 전시기획 규칙 파일을 만들면:

```markdown
# .cursorrules — 전시기획 프로덕션 시스템

이 프로젝트는 몰입형 전시를 기획·제작하는 작업이다.

## Your Role
전시기획의 프로덕션 시스템으로 기능한다.
공간 조건 + 주제를 받으면 아래를 생성:
1. 리서치 브리프 (유사 사례 5건 + 트렌드)
2. 컨셉 스테이트먼트 (1문장, 대비 구조)
3. 공간 계획서 (동선, 장비 배치, 조도)
4. Midjourney 프롬프트 세트 (컨셉 시각화 5장)
5. 프레젠테이션 스크립트 (3분)
6. 예산 개요 (장비 + 시공 + 운영)

## 공간 연출 규칙
- 관객 동선이 내러티브를 형성
- 조도 기본값: 150lux (몰입형), 300lux (전시형)
- 프로젝터 배치 공식: [수량], [위치], [투사거리], [해상도]
- 사운드: 최소 4채널 서라운드
- NEVER: 형광등 직접 노출, 백색 벽면 그대로 사용, 과도한 안내 사인

## 프레젠테이션 리듬
- 슬라이드 01: 컨셉 1장 이미지 + 스테이트먼트
- 슬라이드 02: 공간 평면도 + 동선
- 슬라이드 03: 장비 배치도 + 렌더링
- 슬라이드 04: 예산 요약 (1페이지)
- 슬라이드 05: 일정 (1페이지)

## Tone
- 구체적, 수치 기반
- 감성 서술은 컨셉 스테이트먼트에서만
- 예산은 반드시 근거 포함 (시장 단가 참조)
```

### 에이전트/규칙 파일을 AI에게 만들어달라고 하는 방법

```
에이전트 파일 만들기:

① AI에게 말한다:
   "내 촬영 프로젝트용 에이전트를 만들어줘.
    - 역할: 촬영 계획 수립 + 결과물 심사
    - 입력: 클라이언트 브리프
    - 출력: 촬영 계획서, 체크리스트
    - 규칙: 과거 촬영 기록을 참조할 것
    - 형식: Claude Code 에이전트 (.claude/agents/ 폴더용)"

② AI가 에이전트 파일을 생성한다.

③ 테스트: 실제 브리프를 넣고 결과를 확인한다.

④ 규칙 수정: "결과물 심사 시 클라이언트 톤 체크 항목 추가해줘"
   → AI가 에이전트 파일 수정

멀티 에이전트 만들기:

① "이 촬영 디렉터 에이전트 외에, 클라이언트 시뮬레이터랑
    스타일 일관성 검사기도 만들어줘"

② AI가 3개 에이전트 파일을 생성한다.

③ "이 3개를 파이프라인으로 연결해줘.
    촬영 계획 → 실행 → 클라이언트 시뮬레이션 → 스타일 체크"

④ 파이프라인 완성.
```

## 요약 — 전체 시스템 구조도

```
[규칙 파일]
  나는 누구인가 + 작업 스타일 + 금지 사항
  = AI의 "기본 성격" 설정
        │
        ▼
[에이전트들]
  ┌─────────┬──────────┬──────────┬──────────┐
  │ 제안서   │ 견적서    │ 리서치    │ SNS      │
  │ 작성기   │ 생성기    │ 요약기    │ 매니저    │
  └────┬────┴────┬─────┴────┬─────┴────┬─────┘
       │         │          │          │
       └─────────┴──────────┴──────────┘
                      │
                      ▼
[파이프라인]
  리서치 → 분석 → 문서 생성 → 검증
  = 여러 에이전트를 순서대로 연결
                      │
                      ▼
[옵시디언 위키]
  모든 결과를 축적 → AI가 패턴 발견
  = 시간이 쌓일수록 시스템이 똑똑해짐
```

```
AI 활용의 5단계:

1단계: 대화      → 웹에서 질문/답변 (문서 1)
2단계: 파일 조작  → 설치형으로 내 파일 직접 수정 (문서 1)
3단계: 도구 연결  → MCP로 외부 서비스 연결 (문서 1)

5단계: 시스템화   → 규칙 파일 + 에이전트 + 파이프라인 (문서 4 ← 여기)

각 단계는 이전 단계를 포함한다.
5단계에 있는 사람은 1–4를 전부 쓰고 있다.
```

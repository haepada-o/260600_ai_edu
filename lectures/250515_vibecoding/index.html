    개발 워크플로우: AI 바이브코딩과 실전 프로토타이핑 (deepsea2 사례)¶

    대상: 비개발자 빌더, 테크-아트 기획자, PM | 시간: 2시간 특강 | 핵심: 코딩 지식 없이도 Cursor, Antigravity 등의 에이전트 도구를 활용하여 비즈니스 로직 설계부터 완성형 백엔드/모바일 앱 프로토타입까지 초고속으로 빌딩하는 실전 워크플로우.

    📌 바이브코딩 프로토타이핑 로드맵 (4단계)¶

    이 가이드는 실제 승아님이 AI 에이전트를 조율해 deepsea2 및 테크-아트 모바일 앱 프로젝트를 빌딩했던 사례를 기반으로 설계된 바이브코딩 입문서입니다.

    Step 1. 설계도와 비즈니스 로직 먼저 격리하기 (20분)¶

    • 판단 (Trigger): AI 에이전트(Cursor 등)에게 다짜고짜 "코딩해줘"라고 해서 산더미처럼 꼬여버린 버그 코드를 받고 좌절할 때.
    • 할 일 (Actions):
    • 구현에 들어가기 전, 인간의 언어로 프로젝트 사양서(Spec)를 마크다운 파일로 명확히 작성한다.
    • 어떤 데이터베이스 테이블이 필요한지, 어떤 화면 흐름(Flow)이 구성되는지, API의 엔드포인트는 무엇인지 먼저 문자로 확정한다.
    • 처음이라면 (Tip):
    • 개발자가 없더라도 걱정하지 마십시오. AI에게 아래 프롬프트를 사용하여 기획 문서를 시스템 아키텍처 문서로 먼저 파싱하게 만드십시오. 
      [기획 방향]: [내 서비스 한 줄 아이디어]
이 기획을 기반으로 비개발자도 이해할 수 있는 전체 폴더 구조, 필요한 API 엔드포인트 목록, 그리고 비즈니스 로직 시나리오 3개를 설계해줘. 코드는 작성하지 마.

    Step 2. AI 에이전트와 페어 프로그래밍 가동 (40분)¶

    • 판단 (Trigger): 시스템 설계가 확정되었고, 실제 코드 작성(Python, Dart, Flutter 등)을 에이전트에게 위임할 때.
    • 할 일 (Actions):
    • 기획/설계 문서를 프로젝트 폴더 내에 배치하고 에이전트에게 참조시킨다.
    • 코드를 한꺼번에 짜 달라고 하지 말고, 가장 핵심이 되는 백엔드 유틸리티 함수나 모델 정의부터 순차적으로 요청한다.
    • 에이전트가 코드를 작성하는 도중, 뻔한 템플릿 코드나 에러를 발생시키는 방식은 과감히 반려한다.
    • 실전 프롬프트 패턴:
    • "~ 라이브러리 사용은 금지하고 내장 모듈 위주로 가볍게 빌드해줘."
    • "이 클래스 정의서의 인터페이스에 맞게 비즈니스 로직 코드만 독립형 모듈로 작성해줘."

    Step 3. 린팅(Linting) 및 에러 로그 역추적 검증 (30분)¶

    • 판단 (Trigger): 에이전트가 생성한 코드가 동작하지 않고 로컬 쉘에서 붉은색 에러 로그가 터져 나올 때.
    • 할 일 (Actions):
    • 에러 메시지 전체를 숨기지 말고 그대로 터미널에서 복사해 에이전트 창에 던진다.
    • AI가 소설을 써서 짐작하지 못하도록, 에러가 발생한 지점의 상하 30줄 코드와 함께 에러 덤프를 제공한다.
    • 수정 코드를 바로 덮어쓰기 전에, 왜 이 에러가 발생했는지 물리적 원인(원인-해결책)을 요약 보고받는다.
    • 처음이라면 (Tip):
    • 에러가 나면 "고쳐줘"라고만 말하지 마십시오. 아래 구조로 질문하십시오. 
      [에러 메시지]: [터미널 에러 복사]
[의심 코드]: [에러 라인 주변 코드 복사]

왜 이 문제가 발생했는지 논리적 버그를 지적하고, 이를 영구적으로 방지하기 위해 코드를 어떻게 리팩토링해야 하는지 제안해줘.

    Step 4. 결정 사항 잠금(Locking) 및 문서 갱신 (30분)¶

    • 판단 (Trigger): 코드가 정상 작동하기 시작했고, 다음 기능 확장을 위해 현재의 성공적인 빌드 버전을 고정하고 싶을 때.
    • 할 일 (Actions):
    • 성공적으로 빌드된 핵심 아키텍처와 결정 내역을 README.md나 .claude/rules/ 하단에 결정 로그(Decision Log)로 추가하여 잠근다.
    • 새로운 작업을 시작할 때 AI에게 이 결정 로그를 읽게 만들어, 이전의 성공 사례나 코딩 스타일을 유지하게끔 규제한다.
    • 규정 관리 가이드:
    • docs/04_시스템.md 가이드라인의 결정 로그 방식을 참고하여, 코드가 꼬이지 않도록 이력을 엄격하게 박제해 관리하십시오.
